# Skill: SVG Hero Approval Framing

## When to use
When a repository is replacing a code-first README hero with a branded SVG hero and the principal needs an approval-ready proposal rather than the final asset.

## Pattern
Frame approval around the strategic decisions that would cause churn if changed later:

1. **What must land in one glance** — the product truth the hero communicates
2. **What copy supports the image** — usually one or two lines, never a mini-essay
3. **What brand constraints govern execution** — palette, typography, platform safety, restraint
4. **What educational content moves lower** — so the README still teaches after the hero stops carrying raw code
5. **What the principal is actually approving** — message, hierarchy, and concept lock; not pixel-level SVG craft

## Why it works
SVG hero reviews often stall because people argue about rendering details before agreeing on the message. This framing separates **strategy approval** from **execution polish**, which keeps the review about the real decision.

## Applied example
For Precept issue #4, the useful approval ask is not “Do you like this SVG?” It is:

- Should the first screen be visual-first?
- Does the hero need to show governed state flow plus a blocked invalid path?
- Does Subscription Billing remain the temporary live concept?
- Does the raw DSL/C# proof move lower in README?

Once those are settled, the SVG can be refined without reopening narrative strategy.
