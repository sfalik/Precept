10. Visual Surfaces DRAFT

Precept appears on five primary visual surfaces — each serves a distinct user need, but all share the same semantic color vocabulary and typography principles from the locked system. This section describes how the brand system translates into each surface.

Surface 1: Syntax Editor

Purpose: The `.precept` text file in VS Code where developers author domain contracts. Syntax highlighting, semantic tokens, hover cards, diagnostic underlines, and inline constraint indicators.

Primary visual concerns:

  • Scannability at glance — a developer skims down the file and immediately recognizes structural parts (declarations vs. rules vs. data)
  • Constraint visibility — states and events under rule pressure must be visually distinguished without slowing read
  • Diagnostic clarity — error underlines and inline messages must interrupt attention without overwhelming
  • Monospace legibility — code rendering in small size, dark background; every hue and weight choice is functional, never decorative

Color application: Use the complete 8-shade authoring palette (§4) exactly as locked. Syntax highlighting tokens map to the semantic system:

  • Indigo shades (#4338CA semantic, #6366F1 grammar) for structure keywords and grammar — establishes the control backbone of the file
  • Violet (#A898F5) for state names; italic if constrained by `assert`
  • Cyan (#30B8E8) for event names; italic if constrained by `assert`
  • Slate shades (#B0BEC5 names, #9AA8B5 types, #84929F values) for data — three depths separate at a glance
  • Gold (#FBBF24) for message strings in `because` and `reject` — the human explanation earns the warm interrupt
  • Comments (#9096A6, italic) — intentionally muted, outside the semantic palette
  • No runtime verdicts in syntax highlighting. Green/red/yellow are reserved for runtime outcomes only; they never appear in the editor, so authoring-time color cannot be confused with runtime verdict

Typography application: Cascadia Cove monospace, standard rendering sizes (13–14px in VS Code). Typography is the second signal channel alongside color:

  • Bold: Structure semantic keywords — `precept`, `field`, `state`, `event`, `invariant`, etc. Weight signals "this word drives behavior"
  • Italic: Constrained states, constrained events, and invariant-guarded field names. Italic says "this token is under rule pressure"
  • Normal weight: Structure grammar, unconstrained actors, data types, values, and rule message text

Accessibility notes:

  • Color contrast: All syntax colors meet WCAG AA (4.5:1) against the background (#0c0c0f). Verified in brand-decisions.md color table (CR column)
  • Italic as non-primary signal: Italic marks constraint pressure as a supplement to color, not the primary indicator. A developer with red/green color blindness can still distinguish constrained tokens by their italic rendering

AI-first note: The syntax editor is the first thing an AI agent reads when learning a Precept file. Structured coloring means the agent can parse hue and weight as metadata channels: indigo = control, violet/cyan = actors, slate = data, gold = human text. Semantic tokens enable AI tools like Claude's symbol inspection to understand token kinds without re-parsing. The editor is not just for humans — syntax highlighting is a machine-readable format.

Surface 2: State Diagram

Purpose: The visual graph rendered in the VS Code preview webview, showing states as nodes and transitions as labeled edges. Developers use the diagram to verify logic, understand event flow, and catch impossible paths. AI agents use diagrams to reason about reachability and state coverage.

Primary visual concerns:

  • Visual hierarchy of state roles — initial, intermediate, and terminal states must be instantly distinguishable by shape and color
  • Event labeling — transition arrows must clearly show which event fires the edge, and conditional transitions need guards/assertions visible
  • Layout legibility — node spacing, edge routing, and label placement must prevent visual tangles even with 10+ states
  • Runtime outcome visualization — during inspection, nodes may highlight to show enabled/blocked state transitions

Color application: State diagram coloring uses the same hue families as syntax highlighting but applies them differently — color and shape together encode state role (§8 / §9 of this spec):

  • State nodes: Use the structural indigo family (#A5B4FC initial, #94A3B8 intermediate, #C4B5FD terminal). Lifecycle role is shown by shape, not lifecycle tint. One state hue keeps the diagram clean
  • State names (text): Violet (#A898F5) — matches syntax highlighting, so readers recognize the name immediately
  • Transition arrows: Indigo (#6366F1) — flow belongs to the structure grammar family
  • Event labels on edges: Cyan (#30B8E8) — matches syntax highlighting event hue; label color identifies the event type without re-reading
  • Event sub-shading: If diagram rendering includes event subtype (transition/conditional/stationary), use the cyan sub-shades (#38BDF8, #7DD3FC, #0EA5E9 respectively, per §9)
  • Runtime verdict overlays: When showing live state during inspection, use the verdict colors: Enabled (#34D399), Blocked (#F87171), Warning (#FDE047). These are strictly runtime, never authoring-time
Why no lifecycle tints in diagrams: The compiler cannot know whether a terminal state is "good" (Approved) or "bad" (Rejected). Using green for some terminals and red for others would require manual annotation and create false confidence. Lilac works for all terminals because it signals finality, not valence. Verdict colors stay runtime-only.

Typography application: Cascadia Cove, normal weight for all labels (bold and italic not used in diagrams):

  • State names: 12–13px, centered on node, violet text. Legible even at small zoom
  • Event labels: 11–12px, positioned above or below edge, cyan text. Label placement must avoid edge occlusion
  • Guard/assertion markers: If conditional transitions show their guards, use 10–11px gold (#FBBF24) text, positioned on or near the edge

Accessibility notes:

  • Color + shape redundancy: State roles use both color AND shape (circle, rounded rect, double-border). A color-blind viewer still recognizes initial vs. terminal by shape alone
  • Edge clarity: Transition arrows must have sufficient stroke weight (2–2.5px) and contrast (#6366F1 arrow on dark background). Event labels must be readable at typical extension webview zoom levels

AI-first note: Diagrams are not just for human visual intuition — they are structured graph data that AI agents parse to understand state reachability. Clear label positioning and consistent edge coloring mean an AI tool can extract "which events can fire from State X?" without ambiguity. The diagram layout itself is machine-readable: node positions, edge directions, and labeled transitions form a knowledge graph that supports AI reasoning about coverage, cycles, and dead states.

Surface 3: Inspector Panel

Purpose: The data/field inspector in the preview webview, showing live instance state during event firing. Developers inspect field values, constraint violations, and the effect of transitions. The panel is where data becomes concrete — where abstract rules meet runtime reality.

Primary visual concerns:

  • Field visibility and hierarchy — which fields are set, which are null/default, which are guarded by invariants
  • Constraint feedback — if a field violates an invariant, the violation must be obvious without requiring text parsing
  • Event argument preview — when firing an event, the panel shows what arguments are available and their values
  • State transition clarity — before and after state, with visual indication of which fields changed

Color application: Inspector uses the data and verdict color systems:

  • Field names: Slate (#B0BEC5), normal or italic if guarded by invariant. Same as syntax highlighting so the reader recognizes the field
  • Field types: Slate (#9AA8B5). Type info is secondary to the name and value
  • Field values: Slate (#84929F) for normal values, or verdict colors if a constraint is violated
  • Current state label: Violet (#A898F5) — matches syntax highlighting and diagram state naming
  • Constraint violation indicators: Blocked (#F87171) for invariant violations, Warning (#FDE047) for unmatched guards. Verdict colors signal runtime constraint feedback
  • Enabled indicators: Enabled (#34D399) for valid state, successful transitions, or non-violated fields
  • Read-only field indicators: Dusk indigo (#9096A6) or a neutral shade for fields that cannot be modified directly (computed, state-derived)

Typography application: Cascadia Cove monospace, 12–13px for field names and values:

  • Field name: Normal weight, 12px, left-aligned. Field names that are invariant-guarded can be italic
  • Field value: 12px, monospace to match code appearance. Wider line-height (1.6–1.8) for readability in a tabular layout
  • Constraint message: 11px, gold (#FBBF24) text below the violated field. Message text is the human explanation, so it earns color emphasis and smaller size to distinguish from data values

Accessibility notes:

  • Color + icon redundancy: Constraint violations use both color (Blocked red) and an icon (✗ or warning symbol). A color-blind developer still sees the symbol
  • Table structure: If inspector data is rendered as a table (field name | type | value), use proper `` headers and `/` semantics so screen readers announce the structure

AI-first note: The inspector panel is JSON-serializable state data. An AI agent reading the inspector output needs to understand: current state, current field values, which constraints are violated, and what the next transition would do. Structured layout with consistent field ordering and color-coded constraint feedback means the agent can parse violations without string matching on error text. The inspector is both a human debugging tool and an API contract for AI tools.

Surface 4: Docs Site

Purpose: The documentation website (future surface) where developers learn Precept concepts, read language design docs, explore sample files, and navigate API reference. The docs site is where brand positioning comes to life — it must teach the "why" alongside the "how."

Primary visual concerns:

  • Hierarchy and scanability — tutorials, reference, samples, and theory must be visually stratified so a developer finds what they need instantly
  • Code sample rendering — `.precept` files embedded in docs must use the same syntax highlighting colors as the editor, so readers feel continuity
  • Conceptual callouts — "this is a precept" (small caps), "this is a pattern" (styled differently from narrative prose)
  • Responsive layout — docs must work on desktop (code examples side-by-side with prose) and mobile (stacked layout)

Color application: Docs site maintains the dark-mode semantic palette but may introduce additional neutral/gray shades for navigation, backgrounds, and UI chrome:

  • Code blocks embedded in prose: Use the full syntax highlighting palette — indigo for structure, violet for states, cyan for events, slate for data, gold for messages. Readers see the same colors as in their editor
  • Brand color accents: Navigation, headings, and link highlights use deep indigo (#6366F1). This keeps the brand color family visible throughout the site without overwhelming
  • Callout boxes: Locked brand decisions, conceptual definitions, and "Precept principle" statements may use subtle indigo background (current practice in brand-spec.html callout style)
  • Verdict colors in diagrams and reference tables: If diagrams are embedded in docs, use the same color system as the preview webview (§2)
  • Neutral grays for structure: Navigation backgrounds, borders, table headers, and dividers use the muted palette (#27272a, #18181b, #0c0c0f) to avoid competition with semantic colors

Typography application: Cascadia Cove for code, primary brand font for headings and UI:

  • Page headings (H1): Cascadia Cove 700, small caps, 0.1em letter-spacing — same wordmark treatment as README and brand-spec
  • Section headings (H2, H3): Cascadia Cove 600–700, normal case. Hierarchy through size and weight, not typeface
  • Prose: System font stack or Cascadia Cove 400, 14–16px line-height 1.6–1.8. Readable for long-form content
  • Code samples: Cascadia Cove 400, 12–13px with 1.8 line-height. Syntax highlighting colors applied

Accessibility notes:

  • Heading hierarchy: H1 → H2 → H3, no skips. Screen readers rely on hierarchy to navigate long docs
  • Code sample alt text or captions: Embedded `.precept` samples should have a descriptive caption or nearby prose explaining the concept they demonstrate

AI-first note: The docs site is where AI agents gather context before writing Precept. Claude agents reading the tutorial section learn syntax and patterns. Reference docs with clear structure (headings, code blocks in `

`, definition lists) are easier for agents to parse. Embedded `.precept` samples double as training data — the agent can read a sample and a caption and learn a pattern. Docs site design matters as much for AI comprehension as human comprehension.

Surface 5: CLI / Terminal

Purpose: Command-line output, build logs, diagnostic messages, and error reporting. Developers see this surface when running `dotnet build`, invoking the language server, or checking `precept` command-line tools. This is where the brand system meets terminal constraints.

Primary visual concerns:

  • Error signal clarity — a developer must instantly see "this is an error" without reading prose
  • Terminal color support — many terminals support 256-color or true color; output must be legible in both dark and light terminal themes
  • Structured output — diagnostic messages, file paths, line numbers, and error text must separate visually even in monospace
  • No ambiguity with success/failure — the verdict colors (green/red/yellow) must be reserved for actual success/failure signals, not decorative accents

Color application: CLI output uses verdict colors strategically and sparingly:

  • Success messages: Enabled (#34D399) — "Build succeeded" or "✓ All checks passed"
  • Error messages: Blocked (#F87171) — "Error: Invalid syntax at line 12" or "✗ Constraint violated"
  • Warnings: Warning (#FDE047) — "Warning: Event 'Activate' has no guard; all paths succeed"
  • Neutral output (file paths, context info): Default terminal color (no color applied). The terminal's default foreground color is most readable across theme variations
  • Emphasis (bold text): File names, identifiers, and code elements can use bold or `backtick quoting` to stand out. Avoid color for emphasis on the CLI
  • No structural indigo on CLI. Deep indigo (#6366F1) does not render well on light terminal themes and may be illegible. Reserved for UI only

Typography application: Monospace rendering, standard terminal fonts:

  • Error/warning headers: Bold + verdict color, e.g., `✗ ERROR:` (bold red) followed by the message
  • File path and line number: Normal weight, default color, e.g., `file.precept:12:5:` (no color formatting)
  • Diagnostic message: Normal weight, default color. If a message quotes code, use backticks: `\`FieldName\`` or double-quotes: `"message text"`
  • Suggestion or hint: Dim/muted color if the terminal supports it; otherwise, normal color with a `→` prefix

Accessibility notes:

  • Color + symbol redundancy: Errors use both color (red) and a symbol (✗). Warnings use both color (yellow) and a symbol (⚠ or ⚡). A developer with red/green color blindness still sees the symbol
  • Light terminal compatibility: Verdict colors (#34D399, #F87171, #FDE047) are calibrated for dark backgrounds. For light terminals, either detect theme and swap to darker shades, or remove color and rely on symbols only

AI-first note: CLI output is often captured by AI tools and parsed as structured data. JSON output (e.g., `--json` flag) is machine-readable; human-readable error messages should be parseable too. Consistent error formatting (error symbol + colon + error code + message) means an AI agent can regex-parse failures. Color codes in terminal output can be captured or stripped by tools, so color should never be the only signal — symbol and text structure matter more.

Integration Notes

These five surfaces form the complete visual ecosystem for Precept:

  1. Syntax Editor: Where rules are written. Color encodes structural meaning; typography signals constraint pressure. The gateway into Precept.
  2. State Diagram: Where logic is visualized. Color + shape encode state roles; labels show events. The cognitive map developers build in their head.
  3. Inspector Panel: Where data is debugged. Verdict colors show constraint violations; data colors show field types and values. The moment theory meets reality.
  4. Docs Site: Where concepts are learned. Embeds the same color system as the editor so syntax examples feel familiar. The destination for "how do I...?" questions.
  5. CLI / Terminal: Where builds are reported. Verdict colors + symbols report success/failure; neutral colors for context. The automation surface.

All five surfaces speak the same semantic language: indigo = structure, violet = states, cyan = events, slate = data, gold = messages, plus verdict colors for runtime outcomes. A developer who learns the color system in one surface instantly understands it in the others.
