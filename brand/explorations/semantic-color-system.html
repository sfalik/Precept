Precept — Semantic Color System Exploration

Every color means something. Indigo anchors the brand, then semantic roles drive the rest: emerald for enabled transitions, rose for blocked paths, amber for constraints, slate for structural chrome, and lilac for terminal states. A palette built on meaning, not decoration.

3. Syntax Highlighting — Semantic Token Roles

Keywords in indigo. Types in emerald (they define valid shapes). Constraint strings in amber. State names could inherit their diagram color — initial states in indigo, terminal states in emerald/rose.

Option A: Uniform indigo keywords

All keywords one color. Clean and simple. State semantics only in the diagram.

precept LoanApplication field ApplicantName as string nullable field RequestedAmount as number default 0 invariant RequestedAmount >= 0 because "Amount cannot be negative" state Draft initial state UnderReview state Approved state Declined in Approved assert DocumentsVerified because "Must have verified docs" from Draft on Submit -> set ApplicantName = Submit.Applicant -> transition UnderReview

Option B: Semantic state coloring in syntax

State names inherit their diagram color. Constraints in amber. Creates a visual link between code and diagram.

precept LoanApplication field ApplicantName as string nullable field RequestedAmount as number default 0 invariant RequestedAmount >= 0 because "Amount cannot be negative" state Draft initial state UnderReview state Approved state Declined in Approved assert DocumentsVerified because "Must have verified docs" from Draft on Submit -> set ApplicantName = Submit.Applicant -> transition UnderReview
The key insight: because strings in amber create a natural constraint annotation color. When you see amber text in a .precept file, you know it's a rule explanation — the same color that shows constraint warnings in the diagram and diagnostics.

Option C: Structural semantic coloring (compiler-driven)

Three structural roles the compiler can determine automatically: initial, intermediate (has both inbound & outbound transitions), and terminal (no outbound transitions). Terminal is deliberately neutral — no success/failure implication — because "Approved" and "Declined" are both terminals and the compiler can't know which is positive.

Implementation note: This is fully automatic. The compiler already knows which state is initial, which states have no outbound transitions (terminal), and which are intermediate. The language server emits custom semantic token types (preceptState.initial, preceptState.intermediate, preceptState.terminal) and the VS Code extension ships default color mappings. Zero DSL changes. Zero user annotation.
My recommendation: T1 (Lilac) keeps everything in the indigo family — initial is medium indigo, intermediate is slate, terminal is light indigo/violet. The three structural roles form a visual progression within the same hue family, and "Declined" in lilac reads as "finished" not "bad." T2/T3 risk looking like they belong to a different design system. T4 is too close to emerald.

Same typeface family for wordmark and code. The wordmark is Inconsolata 700 small-caps + 0.1em tracking. The code is Inconsolata 400-600 normal case. Zero font-family mismatch — the code literally grows out of the wordmark.
Why small caps works for "Precept" specifically: Small caps are the typographic convention for defined terms — legal codes, philosophical axioms, constitutional clauses. The word "precept" literally means "a strict rule of action." Setting it in small caps isn't a style choice — it's a semantic one. The typography says "this is a defined concept" before you read the word. Combined with Inconsolata's narrow, precise letterforms, it reads like something carved into a specification.

8. Event Coloring — Where Do Events Fit in the Palette?

Events are the verbs of a precept — they trigger transitions, carry arguments, and have their own constraints (on Submit assert ...). The language server currently colors them as "function" (typically yellow/gold in VS Code themes). Should they get a dedicated semantic color in our palette?

Event surfaces in the DSL:

  • event Submit with ... — declaration
  • on Submit assert ... — event-level constraint
  • from Draft on Submit — transition trigger
  • set Name = Submit.Applicant — argument access

precept LoanApplication field ApplicantName as string nullable field CreditScore as number default 0 state Draft initial state UnderReview state Approved state Declined event Submit with Applicant as string on Submit assert Applicant != "" because "Name required" event Approve with Amount as number from Draft on Submit -> set ApplicantName = Submit.Applicant -> transition UnderReview from UnderReview on Approve -> transition Approved
Assessment:
  • E1 (Violet) — Collides with lilac terminal states. "Submit" and "Approved" become the same color, which is confusing since they're fundamentally different concepts (action vs destination).
  • E2 (Gold) — Collides with amber because strings. The on Submit assert ... because "Name required" line has event AND constraint in the same hue.
  • E3 (Sky) — Clean separation. A 6th semantic color, but it carries clear meaning: "this triggers change." Cool cyan reads as action/signal. State = nouns (indigo/slate/lilac), Event = verbs (sky), Data = variables (light blue #9CDCFE), Rules = constraints (amber).
  • E4 (Default) — Simplest. No new color. But events are arguably the most important identifiers in a precept — they're the API surface. Leaving them unstyled loses semantic information.

The semantic case for sky: A precept has three kinds of named things — states (nouns, where you are), events (verbs, what happens), and fields (data, what you know). If each gets its own semantic register, the code becomes instantly scannable: indigo/slate/lilac = states, sky = events, light blue = fields. You can find the verbs at a glance.
