Precept — Semantic Color System Exploration

SUPERSEDED — Palette Locked

This exploration was the starting point for the semantic color system. The final converged palette is locked in brand-spec.html §4 Color System and brand-decisions.md. Key changes from this exploration:

  • States: Indigo/Slate/Lilac tiers → Violet 260° (#8B7AED / #A898F5 / #C0B4F8)
  • Events: Lilac/Gold/Sky candidates → Cyan 195° single shade (#30B8E8)
  • Data: Emerald types + light-blue vars → Bright Slate 215° italic (#B0BEC5 / #9AA8B5 / #84929F)
  • Structure: Single indigo → 3-tier Indigo semibold (#4338CA / #4F46E5 / #6366F1)
  • Rules: Amber because → Gold 45° (#B8860B keywords + #FBBF24 messages)
  • Typography: Added semibold for structure, italic for data

Every color means something. Indigo anchors the brand, then semantic roles drive the rest: emerald for enabled transitions, rose for blocked paths, amber for constraints, slate for structural chrome, and lilac for terminal states. A palette built on meaning, not decoration.

3. Syntax Highlighting — Semantic Token Roles

Keywords in indigo. Types in emerald (they define valid shapes). Constraint strings in amber. State names could inherit their diagram color — initial states in indigo, terminal states in emerald/rose.

Option A: Uniform indigo keywords

All keywords one color. Clean and simple. State semantics only in the diagram.

precept LoanApplication field ApplicantName as string nullable field RequestedAmount as number default 0 invariant RequestedAmount >= 0 because "Amount cannot be negative" state Draft initial state UnderReview state Approved state Declined in Approved assert DocumentsVerified because "Must have verified docs" from Draft on Submit -> set ApplicantName = Submit.Applicant -> transition UnderReview

Option B: Semantic state coloring in syntax

State names inherit their diagram color. Constraints in amber. Creates a visual link between code and diagram.

precept LoanApplication field ApplicantName as string nullable field RequestedAmount as number default 0 invariant RequestedAmount >= 0 because "Amount cannot be negative" state Draft initial state UnderReview state Approved state Declined in Approved assert DocumentsVerified because "Must have verified docs" from Draft on Submit -> set ApplicantName = Submit.Applicant -> transition UnderReview
The key insight: because strings in amber create a natural constraint annotation color. When you see amber text in a .precept file, you know it's a rule explanation — the same color that shows constraint warnings in the diagram and diagnostics.

Option C: Structural semantic coloring (compiler-driven)

Three structural roles the compiler can determine automatically: initial, intermediate (has both inbound & outbound transitions), and terminal (no outbound transitions). Terminal is deliberately neutral — no success/failure implication — because "Approved" and "Declined" are both terminals and the compiler can't know which is positive.

T2/T3 risk looking like they belong to a different design system. T4 is too close to emerald.

Assessment:
  • E1 (Violet) — Collides with lilac terminal states. "Submit" and "Approved" become the same color, which is confusing since they're fundamentally different concepts (action vs destination).
  • E2 (Gold) — Collides with amber because strings. The on Submit assert ... because "Name required" line has event AND constraint in the same hue.
  • E3 (Sky) — Clean separation. A 6th semantic color, but it carries clear meaning: "this triggers change." Cool cyan reads as action/signal. State = nouns (indigo/slate/lilac), Event = verbs (sky), Data = variables (light blue #9CDCFE), Rules = constraints (amber).
  • E4 (Default) — Simplest. No new color. But events are arguably the most important identifiers in a precept — they're the API surface. Leaving them unstyled loses semantic information.

Collision & Gap Analysis

Where does the current color assignment break down?

Collisions
  • Event names and Precept name are the same yellow #DCDCAA — are they the same concept? precept LoanApplication vs event Submit
  • Fields and event args share #9CDCFE — intentional? Both are "data you can read in expressions," so arguably yes.
  • If E1 (violet) is chosen for events: Submit and Approved become identical — action vs destination collision.
  • If E2 (gold) is chosen: Submit assert ... because "Name required" — event and constraint string merge.
Clean Separations
  • Keywords (all indigo) vs Types (emerald) — zero ambiguity.
  • State names have 3-tier structural coloring — always distinguishable from fields, events, keywords.
  • Constraint strings (amber) stand alone — nothing else is amber.
  • Numeric literals and operators are neutral — never confused with identifiers.

The design question: Should events and fields get branded palette colors (adding sky/cyan for events, keeping or branding #9CDCFE for fields)? Or do they stay as default theme tokens, only states getting structural treatment?

Arguments for branding: Events are the API surface — the most important identifiers for consumers. Fields carry all the business data. Leaving them in default theme colors means 60% of identifiers are unbranded.

Arguments against: VS Code users already recognize #DCDCAA as "function" and #9CDCFE as "variable." Overriding those expectations may cause cognitive friction. The existing defaults are already semantically distinct from each other and from our keywords.

What changed from the previous palette:
  • States → Violet family (#C084FC / #A78BFA / #D8B4FE). Previously indigo/slate/lilac — three unrelated colors pretending to be variants. Now a true family: vivid → medium → soft within one hue range. States are "places in the Precept machine," and violet is indigo's analogous neighbor — they're part of the same world.
  • Events → Sky #38BDF8. Previously default VS Code yellow #DCDCAA. Now a dedicated category color. Sky blue pops against both indigo keywords and violet states. Events are the API surface — they deserve to be instantly findable. Blue reads as "active, precise, signal."
  • Data → Slate #94A3B8. Fields, types, literals, args — all unified as "the information layer." Quiet and recessive, letting structure/states/events carry the rhythm.
  • Constraints → Not a category. Previously amber was treated as Dimension 1. Now amber is purely a verdict color (Rule) that appears in syntax through because/reject strings. Constraints produce verdicts about other things — they don't exist independently.
  • Emerald → Pure verdict. Previously double-duty (types in syntax + success in runtime). Now types are slate (Data category). Emerald is reserved entirely for the Enabled verdict — diagrams, inspector, diagnostics. Cleaner separation.
Open question: Does the data category need internal variation the way states do? Currently everything from string to CreditScore to 0 to true is the same #94A3B8. That groups them conceptually (all "information") but may feel flat. Options: (a) keep flat — the indigo keywords create enough rhythm, (b) use a brighter slate for type keywords and a dimmer one for literals, (c) let numbers keep a slight warmth.
