Precept — Visual Language Exploration

The DSL syntax and state diagrams ARE the visual identity. The icon is one element in a broader system that includes .precept files, state diagrams, diagnostics, and the inspector panel. Background: #0c0c0f. Font: Inconsolata. Palette: 17 shades, 6 families.

Locked decisions applied: All examples below use the locked 17-shade semantic palette (indigo structure, azure states, sky events, slate data, gold rules, emerald/coral/yellow verdicts), Inconsolata font, #0c0c0f background, and compiler-driven structural coloring for both states and events. See brand-decisions.md and brand-spec.html for canonical references.

1. The .precept file IS the hero visual Locked

Like Prisma's .prisma schema or Tailwind's utility classes, the DSL syntax is the most frequently seen brand surface. Every keyword-anchored line creates a distinctive left-edge pattern. The code IS the hero image.

loan-application.precept

Four sections, one specification. The data it carries, the phases it moves through, what can happen, and how it behaves. You are not writing code — you are describing how the workflow should work.

precept LoanApplication field ApplicantName as string nullable field RequestedAmount as number default 0 field CreditScore as number default 0 field DocumentsVerified as boolean default false invariant RequestedAmount >= 0 because "Requested amount cannot be negative" state Draft initial state UnderReview state Approved state Funded state Declined in Approved assert DocumentsVerified because "Approved loans must have verified documents" event Submit with Applicant as string, Amount as number, Score as number on Submit assert Applicant != "" because "An applicant name is required" on Submit assert Amount > 0 because "Loan requests must be positive" event Approve with Amount as number event Decline with Note as string event VerifyDocuments event FundLoan from Draft on Submit -> set ApplicantName = Submit.Applicant -> set RequestedAmount = Submit.Amount -> set CreditScore = Submit.Score -> transition UnderReview from UnderReview on VerifyDocuments -> set DocumentsVerified = true -> no transition from UnderReview on Approve when DocumentsVerified && CreditScore >= 680 -> transition Approved from UnderReview on Approve -> reject "Approval requires verified documents and strong credit" from UnderReview on Decline -> transition Declined from Approved on FundLoan -> transition Funded
The authoring experience is the value proposition. A precept reads top to bottom in natural sections — data, states, events, behavior — each one self-explanatory. Whether you write it by hand or an AI generates it from a conversation, the result is immediately readable. The coloring reinforces the sections: you can scan the left edge and know exactly where you are.

Minimum viable snippet — three candidates Exploring

Minimally complete: cycles back, ~10 lines, has data + a gold rule + transitions. Universal domain everyone understands instantly.

A) Thermostat — 10 lines
precept Thermostat field TargetTemp as number default 72 state Idle initial, Heating event SetTarget with Temp as number on SetTarget assert Temp > 0 because "Absolute zero is not a comfort setting" event Reached from Idle on SetTarget -> set TargetTemp = SetTarget.Temp -> transition Heating from Heating on Reached -> transition Idle
B) ParkingMeter — 15 lines
precept ParkingMeter field TimeLeft as number default 0 invariant TimeLeft >= 0 because "The meter lacks a reverse gear" state Empty initial, Active event Feed with Minutes as number on Feed assert Minutes > 0 because "Coins go in, not out" event Check from Empty, Active on Feed -> set TimeLeft = TimeLeft + Feed.Minutes -> transition Active from Active on Check when TimeLeft == 0 -> transition Empty from Active on Check -> no transition
C) Toaster — 9 lines
precept Toaster field Level as number default 3 invariant Level >= 1 && Level <= 5 because "The dial only goes to 5, trust us" state Idle initial, Toasting in Idle edit Level event Start, Done from Idle on Start -> transition Toasting from Toasting on Done -> transition Idle
D) Stopwatch — 9 lines
precept Stopwatch field Elapsed as number default 0 invariant Elapsed >= 0 because "Time only moves forward" state Stopped initial, Running event Start, Stop, Reset from Stopped on Start -> transition Running from Running on Stop -> set Elapsed = Elapsed + 1 -> transition Stopped from Stopped on Reset -> set Elapsed = 0 -> transition Stopped
A) Thermostat — 10 lines. Idle ↔ Heating. Data flow, event assert, explicit events.
B) ParkingMeter — 15 lines. Empty ↔ Active. Invariant + event assert, accumulation, multi-line pipelines.
C) Toaster — 9 lines. Idle ↔ Toasting. Editable Level with invariant, comma-separated events.
D) Stopwatch — 9 lines. Stopped ↔ Running. Start/Stop/Reset on one line, data increment + clear, invariant guard.

2. State diagrams as the accompanying graphic system Locked

The VS Code preview panel renders state diagrams using the same semantic palette as syntax highlighting. Code and diagram share visual language — same azure for states, same sky for events, same verdict colors for edges.

LoanApplication — state diagram

Azure state nodes (origin/active/final shades). Sky event labels. Emerald enabled edges. Coral blocked edges. Gold rule annotations.

DSL + Diagram pairing: The code defines it, the diagram visualizes it. Together they are the visual identity system — code on the left, state diagram on the right. This pairing appears in the README, the VS Code preview panel, and eventually docs. Both use the same palette; the diagram is a direct projection of the code, not a separate design.
What to explore:
  • Node shapes — pill (current) vs rounded rect vs circle for origin? Double-border for final is clear, but is there a cleaner terminal marker?
  • Edge routing — straight lines vs curved? Curved might look more polished but harder to generate programmatically.
  • Constraint annotations — the gold "!" badge works here. Should guards appear as edge labels or separate annotations?
  • Self-loop rendering — the VerifyDocuments arc looks OK. Does it need a different visual when the event is stationary?
  • Layout algorithm — what drives node positioning? Left-to-right flow with lifecycle ordering (origin → active → final)?

5. Visual identity thesis Locked

Precept's visual identity is not an icon — it is an ecosystem of four surfaces:

  1. DSL syntax — keyword-anchored lines, arrow chains, the distinctive left-edge rhythm. Six visual voices (indigo structure, azure states, sky events, slate data, gold rules, verdict colors). This is the hero visual.
  2. State diagrams — azure nodes, sky event labels, verdict-colored edges, gold rule annotations. Generated directly from the precept. The companion graphic.
  3. The icon — a mark that hints at both ideas (inscribed rules + state flow). Used at small sizes everywhere. Currently exploring.
  4. Diagnostic output — PRECEPT-prefixed codes with structured messages, structural state colors inline. The daily-use brand surface.

The icon does not carry the brand alone. The .precept file format and state diagrams do the heavy lifting. The semantic color system is the connective tissue — the same 17 shades appear across all four surfaces.

What is decided:
  • Primary surface = DSL syntax. Secondary = state diagrams. Tertiary = diagnostics. Icon = small mark.
  • Background: #0c0c0f everywhere. Font: Inconsolata (wordmark 700 small-caps, code 400-600).
  • 17-shade semantic palette applied consistently across all surfaces.
  • Compiler-driven structural coloring for states (origin/active/final) and events (transition/conditional/stationary).
Open threads to explore:
  • Inspector panel — the runtime "what can I do from here?" view. Fifth visual surface? Uses the same palette but needs its own layout language.
  • AI/MCP as a visual surface — when an AI agent calls precept_inspect, the JSON output maps to the same categories. Could we render an "AI view" that uses the same palette?
  • README hero layout — final positioning of code + diagram. Side-by-side vs stacked. Image format for og:image.
  • Icon direction — autonomous loop running. Will incorporate locked palette colors once a form converges.
  • Light theme — do we need one? GitHub READMEs can be viewed in light mode. How does the palette adapt?
  • Animation — could the state diagram animate transitions? Fire an event and watch the current-state marker move? This would be compelling for landing pages.
