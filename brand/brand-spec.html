1. Brand Identity

1.1 Positioning & Narrative LOCKED

Primary frame

Category creation — "domain integrity engine"

No competitor occupies this frame. Precept defines the category, then becomes synonymous with it. Same pattern as Temporal ("durable execution"), Docker ("containerization"), Terraform ("infrastructure as code").

Secondary frame

AI-native — "the contract AI can reason about"

Deterministic semantics + structured tool APIs make Precept uniquely suited for AI authoring and validation. This is the "why now" signal, not the primary identity.

Single-sentence positioning

"Precept is a domain integrity engine for .NET — a single declarative contract that binds state, data, and business rules into an engine where invalid states are structurally impossible."

Narrative archetype

Category Creator: "This is a new kind of thing."

Like Temporal's "durable execution" or Tailwind's "utility-first CSS." Requires category education but earns the strongest position.

README narrative arc

  1. One sentence: what it is (resolution)
  2. One sentence: why it's different (implicit conflict)
  3. One code sample (proof)
  4. Feature list or badge row (comprehensiveness signal)
The name "Precept" carries narrative weight — it means "a strict rule of action." The name IS the story.

1.4 Color System LOCKED

8 authoring-time shades in 5 families, plus 3 runtime verdict colors. Comments sit outside that semantic palette as an editorial lane. Background #0c0c0f. Every color has a fixed meaning. Nothing is decorative.

Locked design

Precept syntax highlighting is a dark-mode-only 8-shade system with no user palette overrides. One hue identifies each category; italic signals constraint pressure. States, events, and data names no longer use extra shades to express lifecycle or rule presence. Gold is reserved for human-readable rule messages.

Semantic color as brand

Most tools pick colors for aesthetics, then assign meanings later. Precept's palette is the reverse — every color exists because the compiler knows something about the code and needs a way to say it. The system is organized along two dimensions:

  • Dimension 1 — category: Structure, States, Events, and Data each get a dedicated visual lane so a line is scannable before it is read.
  • Dimension 2 — constraint pressure: Italic means the token is constrained or guarded. Same hue, different posture.
  • Rules are mostly absorbed into structure — assert, reject, when, and no are structural drivers. Gold is reserved for human-language messages.
  • Verdicts stay runtime-only — enabled / blocked / warning remain outside authoring-time syntax, so green and red cannot be mistaken for static code meaning.

Color Usage Q&A

Q: I need a secondary palette highlight. What do I use?

Brand-light #818CF8. It's the indigo accent — use it for hover states, secondary emphasis, or anywhere you need "brand but one step back." If brand-light is still too strong, brand-muted #C7D2FE provides a whisper of indigo for backgrounds and tints.

Q: When does success green appear in the README?

CI passing badges, feature-available checkmarks (✅), and any "this works" indicator. Shields.io badges for build status use color=34D399. Feature lists use ✅ emoji, which renders green naturally.

Q: Is there a role for warning amber in the README?

Yes — beta/preview callouts and "attention needed" badges. Use ⚠️ emoji for inline warnings. For shields.io: color=FCD34D&label=preview. Warning amber signals "this is real but not final."

Q: Is error rose ever used in the README?

No. Error rose #FB7185 is a product UI color — it appears in the inspector when a transition is blocked or a constraint is violated. Marketing copy (README, docs, badges) should never communicate failure. If something doesn't work, fix the product — don't badge it red.

Q: Can I use gold outside of syntax highlighting?

No. Gold #F59E0B is locked to syntax highlighting for because and reject message strings. It exists to make human-readable rule text visually distinct from machine-interpreted code. Using it in UI or badges would dilute that semantic precision.

The README color contract

In a GitHub Markdown context, Precept's brand identity comes through three channels: (1) the SVG wordmark baked in brand indigo #6366F1, (2) the shields.io badge row using #6366F1 as the primary badge color, and (3) the .precept hero code block whose keyword rhythm is recognizable even in GitHub's default syntax theme. Everything else — headings, prose, tables — renders in GitHub's native styles. The brand is unmistakable without fighting the platform.

2.3 Inspector Panel LOCKED

Purpose

Runtime verdict surface. The data/field inspector in the preview webview, showing live instance state during event firing. Developers inspect field values, constraint violations, and the effect of transitions. The panel is where data becomes concrete — where abstract rules meet runtime reality.

Primary visual concerns

  • Field visibility and hierarchy — which fields are set, which are null/default, which are guarded by invariants
  • Constraint feedback — if a field violates an invariant, the violation must be obvious without requiring text parsing
  • Event argument preview — when firing an event, the panel shows what arguments are available and their values
  • State transition clarity — before and after state, with visual indication of which fields changed

Color application

Inspector uses the data and verdict color systems:

  • Field names: Slate #B0BEC5, normal or italic if guarded by invariant. Same as syntax highlighting so the reader recognizes the field
  • Field types: Slate #9AA8B5. Type info is secondary to the name and value
  • Field values: Slate #84929F for normal values, or verdict colors if a constraint is violated
  • Current state label: Violet #A898F5 — matches syntax highlighting and diagram state naming
  • Constraint violation indicators: Blocked #F87171 for rejected events or invariant violations. Verdict colors signal runtime constraint feedback
  • Enabled indicators: Enabled #34D399 for valid state, successful transitions, or non-violated fields
  • Read-only field indicators: Dusk indigo #9096A6 or a neutral shade for fields not declared editable in the current state via in State edit Field

Typography application

Cascadia Cove monospace, 12–13px for field names and values:

  • Field name: Normal weight, 12px, left-aligned. Field names that are invariant-guarded can be italic
  • Field value: 12px, monospace to match code appearance. Wider line-height (1.6–1.8) for readability in a tabular layout
  • Constraint message: 11px, gold #FBBF24 text below the violated field. Message text is the human explanation, so it earns color emphasis and smaller size to distinguish from data values

Accessibility notes

  • Color + icon redundancy: Constraint violations use both color (Blocked red) and an icon (✗ or warning symbol). A color-blind developer still sees the symbol
  • Table structure: If inspector data is rendered as a table (field name | type | value), use proper <th> headers and <tr>/<td> semantics so screen readers announce the structure
  • Formal accessibility audit (color-blind simulation, screen reader testing) is a backlog item

AI-first note

The inspector panel is JSON-serializable state data. An AI agent reading the inspector output needs to understand: current state, current field values, which constraints are violated, and what the next transition would do. Structured layout with consistent field ordering and color-coded constraint feedback means the agent can parse violations without string matching on error text. The inspector is both a human debugging tool and an API contract for AI tools.

2.4 Docs LOCKED

Purpose

Internal documentation artifacts (design docs, tutorials, reference material). These are team-facing documents, not a public-facing documentation site. Where developers learn Precept concepts, read language design docs, explore sample files, and navigate API reference.

Primary visual concerns

  • Hierarchy and scanability — tutorials, reference, samples, and theory must be visually stratified so a developer finds what they need instantly
  • Code sample rendering — .precept files embedded in docs must use the same syntax highlighting colors as the editor, so readers feel continuity
  • Conceptual callouts — "this is a precept" (small caps), "this is a pattern" (styled differently from narrative prose)

Color application

Docs maintain the dark-mode semantic palette but may introduce additional neutral/gray shades for navigation, backgrounds, and UI chrome:

  • Code blocks embedded in prose: Use the full syntax highlighting palette — indigo for structure, violet for states, cyan for events, slate for data, gold for messages. Readers see the same colors as in their editor
  • Brand color accents: Navigation, headings, and link highlights use deep indigo #6366F1. This keeps the brand color family visible throughout the docs without overwhelming
  • Callout boxes: Locked brand decisions, conceptual definitions, and "Precept principle" statements may use subtle indigo background (current practice in brand-spec.html callout style)
  • Verdict colors in diagrams and reference tables: If diagrams are embedded in docs, use the same color system as the preview webview (§2.2)
  • Neutral grays for structure: Navigation backgrounds, borders, table headers, and dividers use the muted palette (#27272a, #18181b, #0c0c0f) to avoid competition with semantic colors

Typography application

Cascadia Cove for code, primary brand font for headings and UI:

  • Page headings (H1): Cascadia Cove 700, small caps, 0.1em letter-spacing — same wordmark treatment as README and brand-spec
  • Section headings (H2, H3): Cascadia Cove 600–700, normal case. Hierarchy through size and weight, not typeface
  • Prose: System font stack or Cascadia Cove 400, 14–16px line-height 1.6–1.8. Readable for long-form content
  • Code samples: Cascadia Cove 400, 12–13px with 1.8 line-height. Syntax highlighting colors applied

Accessibility notes

  • Heading hierarchy: H1 → H2 → H3, no skips. Screen readers rely on hierarchy to navigate long docs
  • Code sample alt text or captions: Embedded .precept samples should have a descriptive caption or nearby prose explaining the concept they demonstrate
  • Light theme: Not planned. Docs are dark-mode-only, matching the editor experience. Light theme support is a backlog item.

AI-first note

The docs are where AI agents gather context before writing Precept. Claude agents reading the tutorial section learn syntax and patterns. Reference docs with clear structure (headings, code blocks in <pre>, definition lists) are easier for agents to parse. Embedded .precept samples double as training data — the agent can read a sample and a caption and learn a pattern. Docs design matters as much for AI comprehension as human comprehension.

2.5 CLI / Terminal ASPIRATIONAL

⚠️ CLI surface is planned; not yet implemented. A standalone precept CLI tool does not currently exist. PRECEPT diagnostic codes (PRECEPT001–PRECEPT054) are surfaced in VS Code's Problems panel via the language server, not in terminal output. dotnet build uses standard MSBuild formatting. This section defines a design contract for a future CLI tool.

Purpose

Command-line output, build logs, diagnostic messages, and error reporting for a future precept CLI tool. When implemented, developers would see this surface when running diagnostics, validation, or compilation from the terminal. This is where the brand system meets terminal constraints.

Primary visual concerns

  • Error signal clarity — a developer must instantly see "this is an error" without reading prose
  • Terminal color support — many terminals support 256-color or true color; output must be legible in both dark and light terminal themes
  • Structured output — diagnostic messages, file paths, line numbers, and error text must separate visually even in monospace
  • No ambiguity with success/failure — the verdict colors (green/red/yellow) must be reserved for actual success/failure signals, not decorative accents

Color application

CLI output uses verdict colors strategically and sparingly:

  • Success messages: Enabled #34D399 — "Build succeeded" or "✓ All checks passed"
  • Error messages: Blocked #F87171 — "Error: Invalid syntax at line 12" or "✗ Constraint violated"
  • Warnings: Warning #FDE047 — "Warning: Event 'Activate' has no guard; all paths succeed"
  • Neutral output (file paths, context info): Default terminal color (no color applied). The terminal's default foreground color is most readable across theme variations
  • Emphasis (bold text): File names, identifiers, and code elements can use bold or backtick quoting to stand out. Avoid color for emphasis on the CLI
  • No structural indigo on CLI. Deep indigo #6366F1 does not render well on light terminal themes and may be illegible. Reserved for UI only

Typography application

Monospace rendering, standard terminal fonts:

  • Error/warning headers: Bold + verdict color, e.g., ✗ ERROR: (bold red) followed by the message
  • File path and line number: Normal weight, default color, e.g., file.precept:12:5: (no color formatting)
  • Diagnostic message: Normal weight, default color. If a message quotes code, use backticks: `FieldName` or double-quotes: "message text"
  • Suggestion or hint: Dim/muted color if the terminal supports it; otherwise, normal color with a → prefix

Accessibility notes

  • Color + symbol redundancy: Errors use both color (red) and a symbol (✗). Warnings use both color (yellow) and a symbol (⚠ or ⚡). A developer with red/green color blindness still sees the symbol
  • Light terminal compatibility: Verdict colors (#34D399, #F87171, #FDE047) are calibrated for dark backgrounds. For light terminals, either detect theme and swap to darker shades, or remove color and rely on symbols only
  • CLI color audit: Current CLI tool color application needs an audit pass to ensure consistency with these principles. This is a backlog item.

AI-first note

CLI output is often captured by AI tools and parsed as structured data. If a future CLI tool provides JSON output, that would be machine-readable; human-readable error messages should be parseable too. Consistent error formatting (error symbol + colon + error code + message) means an AI agent can regex-parse failures. Color codes in terminal output can be captured or stripped by tools, so color should never be the only signal — symbol and text structure matter more.
