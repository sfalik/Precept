# Syntax Coloring Fix Design — Deep Analysis

> **Author:** Frank (Lead/Architect)
> **Date:** 2026-05-12
> **Status:** Design — pending review

---

## 1. Root Cause Analysis

The symptom — "syntax coloring works fine until the language server loads, then it gets messed up" — is caused by **systematic scope mismatches between the TextMate grammar and the language server's semantic token classifications**. The problem is structural, not a one-off bug.

### How VS Code resolves token colors (critical context)

VS Code has two independent colorization layers:

1. **TextMate (TM) grammar** — fires immediately on file open. Regex-based. Assigns TM scope names (e.g., `keyword.declaration.precept`). Colors come from `tokenColorCustomizations.textMateRules` in the extension manifest.

2. **Semantic tokens** — fires after the language server connects. The LS classifies every token by semantic role. Colors come from `editor.semanticTokenColorCustomizations.rules` (written by the extension from a `precept/semanticTokenColors` notification), with a fallback to `semanticTokenScopes` → TextMate scope → textMateRules.

**When semantic tokens arrive, they REPLACE the TextMate classification at each position.** If the semantic token resolves to a different color than the TM grammar assigned, the user sees a visible shift.

### The core problem: two independent classification systems

The Precept extension has:

- A **TextMate grammar** (generated by `tools/Precept.GrammarGen/`) that assigns **context-sensitive** scopes via structural regex patterns — e.g., `keyword.declaration.precept` for any keyword on a declaration line, `keyword.control.precept` for preposition keywords in transition contexts, `storage.modifier.state.precept` for state modifiers.

- A **semantic token lexical pass** (in `SemanticTokensHandler.ProjectLexicalTokens()`) that assigns every keyword its **catalog-fixed visual category** — e.g., `preceptKeywordSemantic` for all declaration/action/outcome keywords, `preceptKeywordGrammar` for all grammar/preposition/connective keywords — regardless of syntactic context.

These two systems make **different classification decisions** for the same token positions.

## 2. Evidence — Specific Mismatches

The TM grammar's structural patterns assign `keyword.declaration.precept` to keywords appearing in declaration context (`precept`, `state`, `event`, `field`, `rule`, `as`, `default`, `ensure`, `initial`). The textMateRule for this scope: **#4338CA, no bold.**

**Files affected:** Every file with state modifier declarations (most `.precept` files).

### RC-3: Transition/scope header keywords change hue

Preposition keywords (`from`, `on`, `to`, `in`) in structural patterns get `keyword.control.precept` (#4338CA). The catalog assigns them `VisualCategory: KeywordGrammar`, so the LS emits `preceptKeywordGrammar` (#6366F1).

| Keyword | Context | TM Color | LS Color | Visible Change |
|---------|---------|----------|----------|---------------|
| `from` | Transition header | #4338CA | **#6366F1** | Hue shift to lighter indigo |
| `on` | Event handler | #4338CA | **#6366F1** | Hue shift to lighter indigo |
| `to` | Entry-gate ensure | #4338CA | **#6366F1** | Hue shift to lighter indigo |
| `in` | State scope | #4338CA | **#6366F1** | Hue shift to lighter indigo |

**Files affected:** Every file with transition rows, event handlers, state-scoped constructs.

### RC-4: `semanticTokenScopes` fallback scope mismatch (secondary)

Even if the primary color rules are present, the `semanticTokenScopes` mapping in `package.json` uses scope names that differ from the TM grammar structural patterns. This is a secondary issue (the primary color resolution path uses `editor.semanticTokenColorCustomizations.rules`), but matters during initial activation before the color notification arrives:

| Semantic Token | `semanticTokenScopes` fallback | TM Grammar actual scope |
|---------------|-------------------------------|------------------------|
| `preceptState` | `entity.name.state.precept` | `entity.name.type.state.precept` |
| `preceptEvent` | `entity.name.event.precept` | `entity.name.function.event.precept` |
| `preceptArgName` | `variable.parameter.arg.precept` | `variable.parameter.precept` |
| `preceptName` | `entity.name.precept` | `entity.name.type.precept.precept` |

The fallback scopes don't match any `textMateRules` entry, so they fall through to theme defaults during the race window.

### RC-5: Built-in token types bypass catalog colors (minor)

The LS emits the standard VS Code semantic token type `"function"` for built-in function calls and `"string"` for typed constants. These standard types get the active theme's semantic colors, which may differ from the TM scopes (`support.function.precept` and `string.quoted.single.precept` respectively). The `BuildColorNotificationPayload` only emits catalog types — it doesn't cover `function` or `string`.

## 4. Options Evaluated

### Option A: `semanticTokenScopes` alignment

**What it does:** Ensure every semantic token type in `semanticTokenScopes` maps to the exact TM scope the grammar assigns.

**Advantages:** Simple, focused on the fallback path.

**Disadvantages:** Doesn't solve the primary problem. The main color resolution path uses `editor.semanticTokenColorCustomizations.rules` (from the color notification), not the `semanticTokenScopes` fallback. The real issue is that the LS reclassifies keywords with different colors/styles than TM assigned — the scopes being misaligned is a secondary issue.

**Permanent fix?** No. The TM grammar assigns context-sensitive scopes (e.g., `keyword.declaration.precept` in declarations) that have no 1:1 semantic token equivalent. You can't map `preceptKeywordSemantic` to both `keyword.other.semantic.precept` AND `keyword.declaration.precept`.

**Changes required:** (a) package.json `semanticTokenScopes`

### Option B: Suppress semantic tokens for TM-covered constructs

**What it does:** Stop emitting semantic tokens for keywords the TM grammar already classifies correctly. Only emit semantic tokens for constructs where they add value: identifiers (field names, state names, event names, arg names, function calls), typed constants, and context-sensitive reclassifications (e.g., `set` in type position).

**Advantages:**
- Eliminates ALL keyword-level override problems (RC-1, RC-2, RC-3) in one change
- Keywords stay colored by TM structural patterns, which are already context-sensitive and correct
- Semantic tokens still fire for identifiers, adding genuine value (distinguishing field refs from state refs from event refs in expression context)
- No race condition for keyword colors — they never change

**Disadvantages:**
- Reduces semantic token coverage — some keywords won't have semantic tokens. This means features that depend on semantic tokens (e.g., semantic highlighting in the editor) won't apply to keywords. In practice, this is fine because the TM grammar already covers keywords exhaustively.

**Permanent fix?** Yes. New keywords added to the catalog automatically get TM grammar coverage (generated). They don't need semantic tokens unless they require context-sensitive reclassification.

**Changes required:** (b) LS `SemanticTokensHandler.ProjectLexicalTokens()`, (a) package.json `semanticTokenScopes` for remaining types

### Option C: `editor.semanticHighlighting.enabled: false` per-language

**What it does:** Disable semantic highlighting entirely for `.precept` files.

**Advantages:** Simplest possible fix. One line in package.json.

**Disadvantages:** Loses ALL semantic token value — no field/state/event distinction in expressions, no constraint-modifier italic styling, no function call highlighting. This is an unacceptable trade.

**Permanent fix?** Yes, but at devastating cost.

**Changes required:** (a) package.json only

### Option D: Fix the LS token type assignments

**What it does:** Make the LS emit context-aware token types that match the TM grammar's context-sensitive classification. For example, emit a `preceptKeywordDeclaration` type for keywords in declaration position, `preceptKeywordControl` for keywords in control position.

**Advantages:** Semantic tokens precisely match TM grammar intent.

**Disadvantages:**
- Requires the LS to replicate TM grammar's context sensitivity, which is complex and fragile
- Doubles the token classification logic (once in TM regex, once in LS)
- New context-sensitive token types would need new `semanticTokenTypes`, `semanticTokenScopes` entries, and color rules — exponential surface growth
- Not derivable from catalog metadata alone — breaks the metadata-driven architecture

**Permanent fix?** No. Fragile and unmaintainable.

**Changes required:** (a) package.json, (b) LS handler, (c) catalog (new token types), (d) theme rules

### Option E: Generator-owned alignment

**What it does:** Extend the grammar generator to also produce `semanticTokenTypes`, `semanticTokenScopes`, and `semanticTokenModifiers` entries for package.json, ensuring they stay in sync with the catalog.

**Advantages:** Keeps catalog as single source of truth for all surfaces.

**Disadvantages:** Doesn't solve the core problem by itself. The issue isn't that the package.json declarations are out of sync — it's that the LS lexical pass emits tokens for keywords that the TM grammar already handles better (context-sensitively). Generating the declarations doesn't fix the runtime behavior.

**Permanent fix?** Addresses the secondary sync problem (RC-4) permanently. Does not address the primary override problem (RC-1, RC-2, RC-3).

**Changes required:** (c) grammar generator, (a) package.json (as generated output)

## 5. Recommended Solution

**Option B (keyword suppression) + Option A (scope alignment for remaining types) + Option E (generator-owned declarations).**

### Core principle

> **The TM grammar owns keyword coloring. The LS owns identifier coloring. Neither steps on the other's territory.**

The TM grammar is already excellent at keyword classification — it uses structural regex patterns that are context-sensitive (declaration vs. control vs. modifier contexts). The LS cannot improve on this without duplicating the context logic. So: **stop emitting semantic tokens for keywords.**

The LS is essential for identifier classification — it distinguishes field names from state names from event names in expression context, where the TM grammar can only assign a catch-all `variable.other.precept` scope. So: **keep emitting semantic tokens for identifiers, typed constants, and context-sensitive reclassifications.**

### Specific changes

#### Change 1: Suppress keyword semantic tokens in the lexical pass

**File:** `tools/Precept.LanguageServer/Handlers/SemanticTokensHandler.cs`
**Method:** `ProjectLexicalTokens()`

Currently, this method emits a semantic token for every lexer token that has a `VisualCategory`. Change it to SKIP tokens whose `Kind` is a keyword/operator token (i.e., tokens with non-null `meta.Text`). Only emit semantic tokens for:

- **Typed constant tokens** (`TypedConstant`, `TypedConstantStart`, `TypedConstantMiddle`, `TypedConstantEnd`) — these need the `string` classification
- **`set` in type position** — this is the one keyword that genuinely needs context-sensitive reclassification (keyword → type)
- **Tokens without `meta.Text`** — these are synthetic/identifier tokens that the TM grammar doesn't cover

The filter is simple: `if (meta.Text is not null && !(token.Kind == TokenKind.Set && IsSetInTypePosition))` → skip.

This eliminates RC-1, RC-2, and RC-3 in one surgical change.

#### Change 2: Align `semanticTokenScopes` with TM grammar scopes

**File:** `tools/Precept.VsCode/package.json` → `contributes.semanticTokenScopes`

Update the fallback scope mappings for identifier types to match the TM structural patterns:

```json
{
  "preceptState": ["entity.name.type.state.precept"],
  "preceptEvent": ["entity.name.function.event.precept"],
  "preceptArgName": ["variable.parameter.precept"],
  "preceptName": ["entity.name.type.precept.precept"],
  "preceptState.preceptConstrained": ["entity.name.type.state.constrained.precept"],
  "preceptEvent.preceptConstrained": ["entity.name.function.event.constrained.precept"]
}
```

Update the catalog `TextMateScope` field to match:

**File:** `src/Precept/Language/SemanticTokenTypes.cs`

```
State  → TextMateScope: "entity.name.type.state.precept"
Event  → TextMateScope: "entity.name.function.event.precept"
ArgName → TextMateScope: "variable.parameter.precept"
Name   → TextMateScope: "entity.name.type.precept.precept"
```

Also update the textMateRules in package.json `configurationDefaults` to include entries for these scopes (most already exist; verify completeness).

This eliminates RC-4.

#### Change 3: Add catalog color entries for built-in token types

**File:** `tools/Precept.LanguageServer/Handlers/SemanticTokensHandler.cs`
**Method:** `BuildColorNotificationPayload()`

Add color rules for the `function` and `string` built-in semantic token types so they get consistent coloring from the color notification instead of falling through to theme defaults:

```csharp
// After catalog entries, add built-in types
new SemanticTokenColorRule("function", "#...", false, false),  // match support.function.precept textMateRule
new SemanticTokenColorRule("string", "#84929F", false, false), // match string.quoted.single.precept textMateRule
```

This eliminates RC-5.

#### Change 4: Generator-owned package.json declarations (future)

**File:** `tools/Precept.GrammarGen/Program.cs`

Extend the generator to also output:
- `semanticTokenTypes` array (from `SemanticTokenTypes.All`)
- `semanticTokenScopes` mapping (from `SemanticTokenTypeMeta.CustomType` → `TextMateScope`)
- `semanticTokenModifiers` array

This ensures the package.json declarations can never drift from the catalog. This is a follow-on improvement, not required for the immediate fix.

## 6. Implementation Plan

### Slice 1: Suppress keyword semantic tokens (eliminates RC-1/2/3)

1. **`SemanticTokensHandler.ProjectLexicalTokens()`** — Add a guard: skip tokens where `meta.Text is not null` unless the token is a typed constant or the `set`-in-type-position reclassification case. This is ~5 lines of code.

2. **`SemanticTokensHandler.BuildLegend()`** — No change needed. The legend still includes all types (they're used by the identifier pass).

3. **Update tests** in `test/Precept.LanguageServer.Tests/` — verify that `ProjectLexicalTokens` no longer emits keyword tokens, and that `ProjectMergedTokens` still emits identifier tokens correctly.

### Slice 2: Align semanticTokenScopes and catalog TextMateScope (eliminates RC-4)

1. **`SemanticTokenTypes.cs`** — Update `TextMateScope` for:
   - `State` → `"entity.name.type.state.precept"`
   - `Event` → `"entity.name.function.event.precept"`
   - `ArgName` → `"variable.parameter.precept"`
   - `Name` → `"entity.name.type.precept.precept"`

2. **`package.json` `semanticTokenScopes`** — Update to match (or regenerate from catalog).

3. **Verify** `tokenColorCustomizations.textMateRules` has entries for all scopes used in `semanticTokenScopes`. Add any missing entries.

### Slice 3: Built-in type color rules (eliminates RC-5)

1. **`SemanticTokensHandler.BuildColorNotificationPayload()`** — Add `function` and `string` entries with colors matching their TM textMateRule equivalents.

2. **`extension.ts`** — Verify the `isPreceptSemanticTokenRuleKey()` function doesn't filter out `function` and `string` entries. Currently it only keeps keys starting with `precept` or `*.preceptConstrained`. This filter would DROP the `function` and `string` entries. **Fix:** either adjust the filter, or use prefixed names (`preceptFunction`, `preceptString`) in the LS legend and color notification, then map via `semanticTokenScopes`.

   **Recommendation:** Use catalog-owned names (`preceptFunction`, `preceptString`) instead of built-in names. This keeps everything under the `precept` prefix and avoids colliding with other extensions. Requires updating `SemanticTokensHandler` to use the catalog names and adding two entries to `SemanticTokenTypes.cs`.

### Slice 4: Generator-owned declarations (future, eliminates drift)

1. **`tools/Precept.GrammarGen/Program.cs`** — Add generation of `semanticTokenTypes`, `semanticTokenScopes`, and `semanticTokenModifiers` JSON fragments, either as separate output files or as sections injected into `package.json`.

2. **Build integration** — Add a build step that regenerates these sections and fails if they differ from the checked-in version (similar to how `tmLanguage.json` is generated).

## Appendix: Architecture Principle

This solution follows the metadata-driven architecture:

- The **catalog** (`SemanticTokenTypes.cs`) remains the single source of truth for token type definitions, including their `TextMateScope` (used for scope alignment) and color information (used for the color notification).
- The **TM grammar** (generated from `Tokens.All` via catalog `TextMateScope`) owns keyword classification.
- The **LS semantic tokens** own identifier classification and context-sensitive reclassification.
- The **grammar generator** can be extended to own all downstream package.json declarations.

The key insight: **semantic tokens should only override TM when they provide information TM cannot.** TM handles keywords perfectly via structural regex. Semantic tokens add value for identifiers (which require semantic analysis to classify). Letting each layer own its strength eliminates the override conflict permanently.
