Event Action Bar — Research-Informed Variants

Five new event action bar variants for the VisitorBadgePickup record form, synthesized from external research across 20+ business products. Each variant is a complete form card at state ReadyForPickup with editable field, Save/Cancel, and both events: Pick Up (active → PickedUp) and Expire (blocked → Expired).

Research Principles Driving v3

  1. Three-tier progressive disclosure: Verdict → Factors → Calculation. Every surveyed business tool that succeeded uses this hierarchy.
  2. "Because" grammar over boolean expressions. Surface Precept's because strings as human sentences, not guard expressions.
  3. Always-visible beats hover/click for primary decision info (NNGroup). If users need it for their main task, show it at rest.
  4. Never hide blocked actions. Show them disabled with a reason. Hiding destroys trust (Jira anti-pattern).
  5. Checklists beat expressions. Guidewire's workplan: concrete items with pass/fail indicators — the most effective pattern.
  6. One-line reason in domain language. Stripe's structured decline codes: reason + next action hint.
  7. IBM Carbon callout: The only public design-system component designed for always-on, non-dismissible contextual explanation.
Variant G Verdict Strip Research-driven

Every event button carries an always-visible one-line verdict underneath. Active events show "→ TargetState", disabled events show the blocking reason in domain language. Zero hover required for the critical information. Based on NNGroup's finding: if users need it for their primary task, show it at rest.

Variant H Checklist Gate Research-driven

Disabled buttons show a tiny inline checklist of guard conditions with colored pass/fail dots. Active buttons show a green checkmark + target state. Like Guidewire's workplan pattern: users see exactly what's required without reading boolean expressions.

Variant I Smart Badge Preview Shane's idea Research-enhanced

Variant J Action Card Research-driven

Each event is a self-contained mini-card — not just a button — containing: event name, status indicator (green/rose dot), target state, and (for blocked events) the reason inline. Active cards are clickable surfaces. Disabled cards show the reason inside their boundary. Like a Stripe payment method card or a Linear issue card — the action itself is a data object.

Variant K Transition Lane Wild card

The research found that no product combines diagram + form + guard explanation in a unified surface. This variant embeds a micro-diagram directly in the action bar: each event is a horizontal lane showing [CurrentState] → EventButton → [TargetState]. The target state reveals on hover. Blocked events show the reason always-visible below their lane. The user reads the transition as a sentence: "From ReadyForPickup, fire Pick Up, arrive at PickedUp."

Recommendation

Ship Variant I (Smart Badge Preview) as the primary event bar pattern.

It is the only variant that satisfies all three of Shane's stated requirements simultaneously: (1) business-app aesthetic (clean button strip, no dev-tool chrome), (2) compact and self-contained (no tables, cards, or lanes expanding the footprint), (3) full inspectability (hover-preview for destination, always-visible reason for blocks).

The badge morph is Shane's own idea — the most natural progressive enhancement because it repurposes a UI element the user already watches. Adding the always-visible blocked reason underneath the disabled button (from the research) closes the one gap in v2's Variant D: blocked reasons are no longer hidden behind hover.

Second choice: Variant G (Verdict Strip) — functionally identical information density but without the badge morph delight. Use G as the fallback if badge animation proves technically complex in the real panel.

Best for multi-guard events: Variant H (Checklist Gate) — the checklist pattern shines when a single event has 3+ guard conditions. Consider H as a progressive-disclosure expansion of I for complex events.
