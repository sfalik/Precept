Semantic Visual System

A Three-Tier Layout For Reading, Scanning, And Proof.

This mock tests a horizontal system for the semantic visual system document rather than rewriting the canonical artifact. The point is to make layout intent legible: editorial sections contract into a centered reading shell, construct reference sections widen into a sidebar plus scanning rail, and specimen pairs expand into a dedicated split surface where authored source can be judged directly against run-time rendering.

Editorial intro

The Semantic World of Precept

Precept governs the integrity of a business entity’s data and configuration under declared rules. The semantic visual system governs how that governed integrity is read. It establishes a stable visual law for Structure, State, Event, Data, and Rule so a reader can move between authored source, diagrams, forms, and explanation without relearning what each construct means.

The contract is continuity, not uniformity. Surface form may adapt to the needs of reading, but construct identity stays fixed. Editorial passages therefore need a narrower shell than the galleries that follow: this tier is where the document makes the argument, states the boundary between authored meaning and run-time verdicts, and sets the expectation that depiction may change while meaning does not.

In the current canonical file, long-form narrative and wide specimen material often live inside the same broad sectional rhythm. This option separates them horizontally so conceptual reading can breathe before the document widens into reference behavior.

Authored meaning

Five authored constructs remain the stable cast: Structure, State, Event, Data, and Rule.

Run-time evaluation

Verdicts and live status report evaluation without replacing the authored construct being read.

Cross-surface contract

The same construct must stay recognizable across definition, state diagram, data form, and explanation.

02 / Core construct

State

Lifecycle coordinate
Modifier logic

State keeps its violet identity while additive signals report initial, terminal, current, constrained, or unreachable status. The modifier never replaces the base construct.

Requested ReadyForPickup PickedUp
Reading promise

A reader should recognize a state name in code, a node in a diagram, and a current-state indicator in runtime as the same semantic thing.

03 / Core construct

Event

Directed action
Base depiction

Events are cyan named actions. They read as clauses in authored source and as directed labels or rows when runtime behavior is rendered.

ApproveRequest PrintBadge Expire
Outcome variants

Live, blocked, unmatched, or rejected event outcomes belong to the runtime layer. They annotate event behavior while event identity stays stable.

04 / Core construct

Data

Governed configuration
Identity

Data names, types, and values form a three-part reading pattern. The field remains the semantic anchor; type and value support that reading without overshadowing it.

VisitorName HostConfirmed OutcomeNote
Operational emphasis

Editable, required, constrained, or violated states belong here because data is what governance most often acts upon.

05 / Core construct

Rule

Constraint language
Base depiction

Rules enter the system as named logic and because/reject language. They read as explicit pressure on what configurations may exist.

because reject assert
Verdict boundary

When a rule fires, the resulting satisfied or violated callout is a runtime verdict overlay, not a new authored construct family.
