Micro State Diagram

The action area of each form card IS a miniature inline state diagram. Current state on the left, events as edge tokens flowing right to their target states — all visible at rest. Every event shows its verdict: why it's available or why it's blocked. No hover required for any critical information. Uses the exact Semantic Visual System CSS for state nodes, edge tokens, and form elements.

Micro State Diagram Concept

Evolved from K4 (Action Lane with Verdict). Shane's direction: "I want it to be a micro state diagram — starting from the current state and only showing the immediate events and next states." The action area reads as a self-contained diagram: [Current] → event → [Target], with verdict lines explaining every decision.

State nodes use exact SVS modifiers: --current (bright violet fill), --terminal (inner double-border via ::after), --initial (filled dot prefix), --unreachable-current (dashed border, dimmed). Edge tokens use exact SVS styling: --active (glow + strong border), --blocked (dashed border).

Scalability Notes

How the micro state diagram pattern holds up from 0 to 5+ events:

  • 0 events (terminal): Clean. The terminal state badge with "no transitions available" text is self-documenting and avoids empty-state confusion.
  • 1 event: Trivially good. Single lane is the most compact possible representation — essentially a one-line summary of what happens next.
  • 2 events: The sweet spot. The canonical VisitorBadgePickup scenario (1 active, 1 blocked) reads naturally as a branching decision. The diverging lanes communicate "two possible paths" at a glance.
  • 3 events: Still clean. Self-transitions introduce visual redundancy (origin = target) but the ↺ glyph resolves it immediately. Active events above blocked events creates a natural priority order.
  • 4–5 events: The practical limit for a single-column vertical list. At 5 lanes, the micro diagram occupies ~280px of vertical space. The information is still scannable, but the action area starts to dominate the card visually. Consider: should events be sorted active-first? Should the diagram get a subtle internal scrollbar at 6+?

Where it breaks: At 6+ events, the vertical list exceeds the natural "action area" mental model. The form starts to look like a diagram with a header, rather than a form with a diagram in its footer. Mitigation strategies:

  • Sort by availability: Active events first, blocked events second. The user sees actionable options immediately without scrolling past blocked reasons.
  • Collapse blocked lanes: Show blocked events as a compact summary row ("3 events blocked") with expand-on-click. Preserves the always-visible principle for active events while managing vertical budget for blocked ones.
  • Horizontal overflow at narrow widths: Long state names (e.g., "WaitingForCustomer") push lanes wider. Minimum viable panel width is ~420px. Below that, state names would need truncation — which harms the diagrammatic quality.

Recommendation: The micro state diagram is strongest at 1–4 events. At 5 events it's acceptable with active-first sorting. At 6+ events, switch to a collapsed-blocked summary to keep the action area from overwhelming the form. For the VisitorBadgePickup precept (max 2 events per state), this pattern is a perfect fit.
