Event Button Affordances

Three approaches solving two gaps in the event bar: no explanation for why a button is disabled, and no pre-click preview of where a transition leads. Shown in a simulated VS Code preview panel. Prototype only — does not modify design/system/semantic-visual-system.html.

Problem 1 — Disabled opacity

A disabled event button gives no feedback about why it is blocked. Only a title tooltip exists — invisible at a glance, absent on touch. A developer needs to know: what has to change before this event becomes available?

Problem 2 — Pre-click destination blind

No-args events fire immediately on click. There is no preview of the target state beforehand — including when the outcome might be a self-transition, a rejection, or a clean state change. For args events, the dialog reveals the target state, but for no-args events the developer must commit blind.

A

Event Row Table

Preferred

Replace the button strip with a compact decision table. Each event is a structured row: event name chip · guard status with reason · target state · fire action. Every piece of information is visible simultaneously in the resting state — no hover, no tooltip, no interaction required.

Does well

All information visible in resting state — zero interaction required. Solves both problems simultaneously: guard reason is always shown for blocked events, target state is always shown for active events. Accessible by default with semantic table markup. Scales to 8–10 events without wrapping chaos. Reads like a runtime truth table — exactly on-brand for Precept's rigor.

Trade-offs

Uses more vertical space than a flat button row. Developers accustomed to a horizontal button strip may need a moment to register rows as interactive. The "Fire" action column needs careful visual treatment — it should feel primary, not like a form submission button. Long guard expressions may need text clamping at narrow widths.

B

Inline Metadata Strip

Runner-up

Preserve the familiar button row but attach a compact one-line metadata strip directly below each button: a target state name for active events, a guard failure reason for disabled events. No interaction required — all affordances visible in the resting state with minimal departure from the current design.

Does well

Smallest departure from the current button design — familiar form preserved. Both problems solved in resting state. Low implementation cost: just add a sibling metadata line per button. Works naturally at narrow widths since the strip wraps with its button. Accessible with proper aria-label and aria-disabled.

Trade-offs

Long guard expressions must be truncated and still risk needing a tooltip as a fallback. Each button gains additional height — with 6+ events wrapping into 2–3 rows, the event dock becomes significantly taller. The strip text competes visually with field labels in the form above it; vertical rhythm needs care.

C

Split Button Disclosure

Runner-up

Active events become split buttons: left zone fires the event, right zone shows the target state name in state color as a fused badge. Disabled events carry a compact ⊗ zone that toggles an inline reason reveal on click or focus — one click to discover, still zero tooltip required.

Does well

Most compact resting footprint of the three approaches — target state rides inline in the same button row at no extra height. The split form communicates intent clearly: left = action, right = consequence. Blocked events are obviously non-passive (the ⊗ badge is interactive), so developers discover the reason in one deliberate click rather than hoping to find a tooltip.

Trade-offs

Blocked event reason still requires an explicit click to reveal — not zero-friction. Two interaction zones in one button require careful keyboard tab and Enter/Space handling to prevent accidental fires. Split button is an unusual pattern in VS Code's native UI vocabulary; some developers may not immediately understand the right-zone affordance without a visual cue for the boundary.

Also Considered — Not Prototyped
D · Rich Hover Popover

On hover, a popover shows full guard expression and target state. No resting footprint. Rejected: hover-only excludes touch, hidden information has high discovery friction for developers scanning quickly.

E · Accordion Expand Row

Click anywhere on a disabled button to expand it into a diagnostic block with full guard details. More informative but requires a click to see anything at all. Better suited as a detail layer within Approach A than as a standalone pattern.

Recommendation — For Shane's review

Approach A — Event Row Table

The table is the right abstraction for the event bar at this product maturity. It frames the event bar as what it actually is: a decision surface that answers "what can happen from here, why, and where does it go?" simultaneously — in the resting state, with no interaction required. Both problems are resolved in a single layout: guard reason is inline for blocked rows, target state is inline for active rows.

Row structure handles 1–10+ events gracefully, is keyboard-navigable, and is accessible by default via semantic table markup. Self-transitions, rejections, and unavailable events each get a distinct visual treatment in the same pattern. The dense, information-first presentation is deeply on-brand for Precept's character as a rigorous runtime tool.

Approach B (Inline Strip) is the right answer if the team wants the smallest possible change to the current button design — it solves both problems with minimal departure and low implementation cost. Approach C (Split Button) is worth pursuing if vertical compactness is paramount, but needs additional design work to handle keyboard accessibility cleanly before it's implementation-ready.
