Precept · Design Exploration

Header Badge Variants

Eleven candidate treatments exploring the form card header. Variants 1–10 explore the top-right badge position, replacing the state pill with richer signals. Variant 11 is a full layout restructure — the micro state diagram becomes the structural spine of the card, not an appendage at the bottom. All shown against the VisitorBadgePickup precept at ReadyForPickup.

Design question

The state pill currently occupies the most prominent badge real-estate in the form card header. What signal deserves that position instead? The current state is already implicit from the micro diagram in the action area — so the header badge can carry richer operational context.

Each variant below shows just the header strip (precept name + record ref on the left, candidate badge on the right). No full form fields — this is a badge-only exploration.

Shows rule health at a glance — how many constraints are currently passing. Green when all rules pass; amber when some are violated. The user sees structural integrity without opening an inspector.

Layout Flip — Full Card Restructure
Layout philosophy: The current state is the card's anchor (top-left) and the event lanes are the destination (bottom-left). The form data sits between them — the user reads top-to-bottom as "I'm in this state, here's the data, here's where I can go next." Precept metadata and Save/Cancel are pushed to the right column, clearing the left spine for the state-diagram flow. The dashed vertical guide line hints at the structural connection between origin and outcomes.

Design Notes

  • Variants 1–4 are data-rich badges — they encode a numeric ratio or proportion. Best when the badge position should carry quantitative context at a glance.
  • Variant 5 is temporal — useful when dwell-time or SLA awareness matters. Requires runtime timestamp data not currently in the DSL.
  • Variant 6 is the most compact at rest (single dot), but relies on hover for the label — breaks the "no-hover-required" principle if the label is essential.
  • Variant 7 is attribution — shows who, not what. Requires actor metadata beyond the current precept model.
  • Variant 8 is forward-looking — primes the user for the primary action. Works best when there's a clear "happy path" event; ambiguous when multiple events are equally likely.
  • Variant 9 is retrospective — visualises history as a mini-sparkline. Useful for audit-oriented workflows; less useful for records with few transitions.
  • Variant 10 is structural — the badge is a gateway, not a signal. It says "more detail is available" without encoding any specific data. Could combine with any other variant.
  • Combinations: Several of these compose naturally. Constraint Pulse (1) + Inspectability Toggle (10) gives health + expandability. Journey Progress (3) + Next-Action Hint (8) gives position + direction.
  • Runtime feasibility: Variants 1, 2, 3, 4, 8, 10 can be derived entirely from the precept definition and current instance state. Variants 5, 7 require runtime metadata (timestamps, actor info). Variant 6 requires a computed risk model. Variant 9 requires transition history.
  • Variant 11 (Layout Flip) is a structural departure — not a badge variant but a full card layout restructure. The micro state diagram becomes the left-hand spine: current state at top-left, event lanes at bottom-left, with the form data sandwiched between. Metadata and actions shift right. This creates a vertical "where I am → what I see → where I can go" reading flow. Requires CSS Grid; no additional runtime data beyond what the micro diagram already needs.
