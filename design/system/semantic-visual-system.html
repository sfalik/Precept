Precept / Design System

Semantic Visual System

The Semantic World of Precept

Precept governs the integrity of a business entity's data and configuration under declared rules. The semantic visual system governs how that governed integrity is read. It defines the construct meaning that must remain legible as a reader moves between authored source, documentation, diagrams, and run-time views.

Five authored constructs carry that meaning: Structure, State, Event, Data, and Rule. Run-time evaluation is expressed through separate verdicts, so a surface can report outcome without redefining the authored construct. The contract is continuity: depiction may adapt to the surface, but the construct stays the same.

Authored meaning Five constructs

Structure, State, Event, Data, and Rule are the authored constructs of the system.

Run-time evaluation Separate verdicts

Foundation Construct Meaning

Each construct carries stable meaning that persists regardless of surface. Structure is always indigo, State always violet, Event always cyan, Data always steel, Rule always amber. Moving from code editor to diagram to data form to timeline, the same construct stays legible even as the medium adapts.

Design-time vs Run-time Views

The design-time view captures what a construct is — its place in the governed contract, independent of any live instance. The run-time view captures what is true of the current configuration now — current state when present, field values, and evaluation results. These two readings are formally distinct. The same construct underlies both; the view changes, the meaning holds. Verdict hues report run-time outcome without becoming new construct colors.

Additive Modifiers & Run-time Overlays

Run-time configuration and evaluation status are expressed as signals layered onto the base depiction — never as replacements. A current state marks the state node; it does not redefine the construct. A violated constraint flags the field row; it does not suppress it. Evaluation output — enabled, blocked, warning — is drawn on top. Overlay reveals; it does not displace.

Cross-Surface Translation

Moving from code editor to diagram to data form to timeline, a construct adopts different visual conventions appropriate to each medium. What remains stable is its meaning: construct color, name form, and the modifier logic that applies to it. Chrome/support tokens stay outside that meaning and handle scaffolding instead, which means a surface can choose a complementary chrome palette without inventing new semantic meaning. The system defines what stays constant so nothing needs to be relearned as surfaces change.

Typography

Typography governs how construct names, prose, and constraint emphasis are read across all five constructs. Geist Mono carries construct names and structural language. Geist carries reading prose. No serifs. Inside the construct colors, italic marks constraint emphasis rather than generic emphasis.

Guideline

No serifs.

Ever. The posture of this document and of Precept's surfaces is architectural, not editorial. Serifs are not part of the system.

Guideline

Mono for everything structural.

If it governs, labels, names, or identifies — it is in Geist Mono. This includes headings. Geist Mono at display size reads as specification, not decoration.

Guideline

Governing keywords are bold.

Semantic indigo keywords render at font-weight 700. Grammar indigo stays normal for connective tokens and logical operators. Color identifies the construct; weight distinguishes governing keywords from connective grammar.

Guideline

Sans for reading prose.

Geist carries paragraphs. It does not appear in construct-name positions, labels, or code. The split is strict. Mono names and governs; sans explains.

Guideline

Italic means constraint emphasis.

Not emphasis. Not decoration. Inside a construct color, italic communicates constraint attention. Outside the construct colors, only comments may use it for editorial sideband context.

Guideline

Every color has a fixed meaning.

Nothing is aesthetic. If a new surface uses color, it must map to an existing semantic color. Local tonal variants may clarify hierarchy inside a color, but they do not create new construct colors. Decorative use of any palette color corrupts the signal.

Guideline

Color says kind. Italic says constrained.

Constraint emphasis is expressed through typography, not extra hues. Constrained states and guarded data names are italic within their own construct color — same hue, italic only.

Guideline

Verdicts are run-time only.

Emerald, coral, and yellow appear only when the engine has evaluated something. They are never used in static surfaces, syntax highlighting, or any authoring-time annotation.

Guideline

Comments are outside the palette.

Dusk #9096A6 in italic keeps comments legible but structurally separate from the five semantic colors. Never repurpose a construct color for editorial annotation.

    Callouts
    notes, tips, cautions, and warnings
  

  

    

      

      

        Note
        
Precept does not evaluate transitions at author time. The Submit event only executes when explicitly fired via the runtime API or MCP tool.

      

    

    

      

      

        Tip
        
Use precept_inspect to preview all available transitions for a given state without modifying any data.

      

    

    

      

      

        Caution
        
Field values passed to precept_fire must match declared field types. Type mismatches are caught at evaluation time, not authoring time.

      

    

    

      

      

        Warning
        
Removing a state referenced by an active transition causes a compile error. All references must be updated before the precept can be re-evaluated.

Defining Precepts

        


        
A precept is a governed schema that defines what a business entity is allowed to become. It declares fields and rules, and when lifecycle matters it may also declare states and events. Everything the entity can do — and everything it cannot — is declared here.


        
        

          
States

          
A state is a named lifecycle position for a stateful precept. When lifecycle matters, states are declared with the state keyword and a name drawn from the State construct color. A new instance begins at the declared initial state; stateless precepts omit state entirely.

          
In a stateful precept, nothing moves between states unless an event fires. This is one expression of Precept's determinism: the same definition, current configuration, and event arguments produce the same outcome.

Events & Transitions

          
Events are the only mechanism that advances state. An event declared under a specific source state can only fire from that state. Attempting to fire it from any other state has no effect  —  Precept does not throw; it simply finds no matching transition.

          
What a transition block does

          
    
            
  • Evaluates all Rule guards in declaration order  —  first failure blocks the whole event
    • 
            
  • Applies all set expressions to produce the next field snapshot
    • 
            
  • Moves the entity to the target state declared by transition
    • 
            
  • Returns the outcome: enabled, blocked, or warning
    • 
          

        


        
        

          

          

            Note
            
Constraints inside a transition block are scoped to that event. Invariants declared at the precept level are evaluated on every transition, regardless of which event fired.

          

        


        
        

          
A minimal precept

          
The following keeps the same LoanApplication scenario used elsewhere on the page so the guide reads as one coherent example across code, diagram, and reference views.

          

            
precept

            
precept LoanApplication
  field  Amount         as number default 0
  field  ApplicantName  as string nullable

  state  Draft      initial
  state  Submitted
  state  Approved
  state  Declined

  event  Submit
  event  Approve
  event  Decline

  from  Draft  on Submit when ApplicantName == null -> reject "Application must include applicant name"
  from  Draft  on Submit when Amount <= 0 -> reject "Application must include positive amount"
  from  Draft  on Submit -> transition Submitted
  from  Submitted  on Approve -> transition Approved
  from  Submitted  on Decline -> transition Declined
