Precept

Semantic Visual System

The semantic visual system defines how Precept means things on screen. It gives structure, states, events, data, explanatory rule language, and runtime verdicts — allowed, blocked, or cautionary — a stable visual grammar so meaning stays recognizable across authoring, topology, preview, diagnostics, and public explanation. The point is not decoration. The point is faster recognition, lower confusion, and a surface language that feels unmistakably Precept.

Meaning stays stable. Semantic identity should survive when the surface changes from code to graph to panel to editorial page.
Forms may adapt. A surface can change shape, density, rhythm, and emphasis without rewriting the semantic contract.
Brand stays visible. Black field, Cascadia hierarchy, indigo structure, and disciplined semantic lanes keep the system recognizably Precept.

Design conviction: this document should be the fullest embodiment of Precept's visual system: rigorous enough to govern the system, expressive enough to prove it. It should dogfood the semantic language it defines so the page feels unmistakably Precept while still functioning as the canonical reference.

Overview

Visual meaning that holds across every surface Precept touches.

Five core constructs, one color contract, plus a separate runtime verdict layer. Every surface built with or on top of Precept should feel like the same language.

Precept should look like a language with internal law and visible confidence. The same semantic grammar applies whether the user is reading code, navigating a diagram, using an inspector, or reading a README. When the surface changes and the meaning holds, the system is working.

A surface is the user-facing context — editor, diagram, inspector, public documentation. A signal is the visual mechanism — color, shape, typography, highlighting, border, or pattern — used within that context to express semantic meaning. The system keeps meaning stable across surfaces by fixing the categories while letting forms adapt to local density, rhythm, and affordance needs.

Signal Language

The canonical map of Precept's signal channels and combination rules.

This section defines which channels are allowed to carry meaning in Precept, what each channel is responsible for, what each channel is forbidden to say, and how those channels compose when the same semantic object moves between editor, diagram, inspector, and documentation surfaces.

Signal language is the contract between semantic meaning and visual treatment. In Precept, the authored construct stays fixed while the surface adapts. A state should still read as a state, an event as an event, data as data, and a rule as explanation even when the presentation shifts from code to diagram to panel to editorial page. Runtime verdicts speak on top of those constructs; they do not replace them.

Core channels: semantic color, typography, layout, and form. These are allowed to carry meaning and are governed here.

Supportive treatments: borders, highlighting, pattern, and motion may reinforce the reading, but they do not get to invent new semantics on their own. If a surface loses color or density, the remaining channels still need to preserve the reading.

Typography

Voice, hierarchy, executable register, and semantic pressure.

Identity and pressure

Separate executable language from reading prose, and mark constrained names without changing which construct they belong to.

Cascadia carries headings, labels, chips, navigation, and executable names. Segoe UI Variable Text carries paragraph explanation. Small caps marks identity. Italic marks constrained or rule-pressured names.

Typography may not replace construct color, runtime verdict, or state-role shape. Italic is not for generic emphasis or decorative editorial flourish.

When density changes, the font split remains. If a surface compresses, keep the register distinction before adding ornament.

Core channel

Layout

Relationship, causality, grouping, and reading order.

Explains what acts on what

Show which parts belong together, what triggers what, and where a result or explanation comes from.

State groups, event rows, transition adjacency, inspector sections, verdict rows, and rule notes should occupy distinct positions in the reading order.

Layout may not reclassify a thing into another construct or rely on density tricks that collapse contract structure, action path, and runtime output into one lane.

If space shrinks, preserve adjacency and order first. Compression is allowed; semantic shuffling is not.

Core channel

Form

Silhouette, object kind, and state-role shape grammar.

Shape before color

Show whether the reader is looking at a shell, state card, event row, transition path, verdict badge, or rule callout.

Precept form includes the outer tablet or frame, equal rounded state cards, an origin-marked initial state, a visibly closed terminal state, directional transition arrows, row-shaped events, panel shells, verdict badges, and inset rule callouts.

Form may not collapse everything into generic cards, pills, or boxes. It may not use verdict shape to replace verdict color, and it may not hide state role behind color alone.

If a surface loses rich treatment, the silhouette ladder still needs to survive: shell, state, event, transition, runtime output, explanation.

Reading order

Construct first, form second, layout third, typography fourth.

The first read is construct identity, the second is object kind, the third is relationship, and the fourth is pressure or explanatory register. A surface can compress those reads; it cannot scramble them.

Composition law

Runtime overlays sit on top of authoring identity.

A state stays a state when it becomes allowed or blocked. Evaluation modifies the reading; it does not replace the underlying family, shape, or role.

Boundary

Brand atmosphere is not signal language.

Black field, tone, contrast, and atmosphere make the system recognizably Precept, but they are not allowed to invent semantic categories. If a treatment cannot survive without aesthetic mood, it is style, not signal.

Foundation Rules

Three rules govern every signal in the system.

Every surface inherits these constraints before it begins solving its own local design problem.

Category permanence

Each authored core construct owns its color.

Structure, states, events, data, and rule language each hold their visual identity across every surface. A surface can shade a color lighter or darker; it cannot reassign that authored construct identity. Verdict colors remain a separate runtime overlay layer.

Shape before color

Structural signals carry meaning before color does.

Whether a state is initial, terminal, or currently active should be legible from shape and position. Color reinforces that reading; it does not replace it.

Runtime only

Verdict colors appear only when the runtime is speaking.

Emerald, rose, and amber signal what happened or what would happen, not what a thing is. They belong to evaluation, not authoring.

Core Constructs

Five core constructs define what a Precept contract is made of.

The authored side of Precept is made of five core constructs. Those constructs determine color, tonal register, and reading contract: structure frames the contract, states name conditions, events signal actions, data carries facts, and rules express constraints and explanation. Verdict is separate: it is the runtime layer that evaluates those constructs without becoming one of them.

Structure Frames scope, grammar, grouping, and governed boundaries.
State Names condition and lifecycle identity without becoming a rainbow taxonomy.
Event Signals action, trigger, and executable intent.
Data Carries field names, types, values, and factual payload as one construct lane.
Rule Interrupts with because-language, rejection-language, and causal explanation.
Runtime layer

Verdict is not a core construct.

Allowed, blocked, and cautionary verdicts are runtime overlays that evaluate authored constructs. They speak after the contract is read; they do not replace structure, state, event, data, or rule identity.

Typography Roles

Type carries identity, interface authority, and semantic pressure.

The split is deliberate and now canonical for this system. The Cascadia stack owns identity and system-facing roles. Segoe UI Variable Text with Segoe UI fallback owns paragraph-length reading. Mono names and governs; the reading face explains.

Cascadia-first authority

Identity and interface chrome stay in the same family.

Wordmark, section headings, labels, chips, navigation, inline semantic tokens, and other system-facing cues should stay on the Cascadia Cove / Cascadia Code / Consolas stack. This keeps the editorial page, the UI shell, and the executable language in one typographic voice.

Reading lane

Paragraph prose should calm the page down.

Use the non-serif reading face for sentences that need sustained comprehension: ledes, section copy, notes, guidance, and public-facing explanation. The current reading lane is Segoe UI Variable Text with Segoe UI and system sans fallbacks. It exists so long-form explanation does not feel like raw source code.

The semantic visual system keeps meaning stable while signals adapt to context. That explanation should read fluently at paragraph length, which is why prose switches to the calmer reading face instead of forcing every sentence through monospace.

Pressure signal

Italic means pressure, not decoration.

Typography is the second semantic channel. Italic belongs to constrained states, constrained events, invariant-guarded names, and nearby editorial annotation about executable material. Do not spend italics on generic emphasis in headings, navigation, chips, or marketing-style prose; that would blur the system's pressure signal.

Composition Primitives

These are the pieces surfaces compose from.

Primitives are not components. They are the semantic parts a surface composes into its own information architecture.

Structural frame

The indigo frame defines context: code block, diagram cluster, overlay shell, or diagnostics group. It tells the user where governed information starts and stops.

Actor pairing

States and events stay visually distinct even when adjacent. The system should never make a user infer whether a label is a condition or an action from wording alone.

Data stack

Data may use tonal variation inside the bright-slate family, but it should still read as one factual lane. Names, types, and values may separate. The family does not.

Rule-message interrupt

Gold enters the interface as explanation the human should notice. It is an interrupt, not a running style. If too much becomes gold, rule meaning disappears.

Constraint pressure

Italic is the shared pressure signal. Surfaces may apply it differently, but they must preserve the meaning: this thing is under explicit rule attention.

Allowed flow

Emerald flow appears when the interface is speaking about a runtime allowance or executable path. It should never be mistaken for general navigation chrome.

Shape & Grammar

How signals are read - shape, scale, and overlay.

These are the rules for reading a signal correctly. Shape and scale determine semantic identity. Verdicts overlay without replacing it.

Context

Use structure to define the governed zone.

Indigo frames, rails, dividers, operators, and headings should establish where the official semantic content begins, ends, and groups itself.

Identity

Keep authored constructs legible at first glance.

State identity must stay distinct from event identity even inside dense surfaces. If a user has to parse the sentence to know which is which, the signal is too weak.

Explanation

Reserve gold for causal language.

Gold belongs to because-language, rejection-language, or other explicit causal explanation. It should not become a general accent lane for headings, decorative dividers, or chrome.

Enabled overlay

Emerald may tint an edge, row accent, badge, or prospective outcome label. The state and event identities beneath it remain violet and cyan.

Blocked overlay

Rose means the path or action is structurally denied in the current runtime context. Use enough geometry, label text, or iconography that the blocked state is not color-only.

Warning overlay

Amber marks unmatched, cautionary, or attention-worthy runtime output. It should read as a diagnostic pause, not as a new semantic family.

Inheritance & Adaptation

Future surface work should start here.

This is the compliance split for future work: what must stay stable, what may adapt, and what every surface must still define locally.

Must preserve

  • Core construct identity for structure, state, event, data, rule language, plus the separate runtime verdict overlay layer.
  • Minimum redundancy rules for blocked, warning, and allowed runtime states.

May adapt

  • Signal medium: shape, weight, chip form, row styling, overlay badge, legend, or pattern.
  • Layout, density, motion, and information grouping for the local workflow.
  • Local tonal tuning within a construct lane when contrast and recognition stay intact.

Must define locally

  • Exact layout, interaction model, and density rules for that surface.
  • Focus, selection, hover, and keyboard behavior that do not collide with semantic cues.
  • Any surface-specific affordances, motion, or help text needed to make the inherited contract usable in context.
Compliance Floor

Rules that apply on every surface.

These rules govern signal application system-wide, not just on individual surfaces.

Preserve semantic identity

A surface may mute or intensify a construct lane locally, but it may not swap authored constructs, demote rule language into generic chrome, or treat verdict colors as authoring identity.

Never ship color-only verdicts

Allowed, blocked, and warning states must use at least one non-hue signal such as iconography, line pattern, label text, badge shape, or row position.

Make current condition explicit

The current state, event availability, and runtime outcome must be named in text somewhere on the surface. Users should not have to infer active condition from framing or tint alone.

Expose the reason when the runtime knows it

If the system knows why an action is blocked, unmatched, or exceptional, the surface must expose that cause as readable copy. Gold remains the preferred explanatory signal when rule language is shown.
