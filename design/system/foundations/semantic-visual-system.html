Precept / Design System

Semantic Visual System

The Semantic World of Precept

Precept governs the integrity of a business entity's data across its lifecycle. The semantic visual system governs how that governance is communicated on screen: how structure, states, events, data fields, and rules each look in every surface where Precept is present.

Five authored constructs. One runtime verdict layer that sits on top of them without replacing them. A stable visual grammar that survives when the context shifts from code editor to topology diagram to data form to public documentation.

02 · Concepts

Five constructs · Two registers · Four channels

The system has four structural parts. Five construct types form the identity substrate — the semantic vocabulary that stays recognizable wherever it appears. Two depiction registers translate that identity into authored source (Definition) and rendered components (Runtime). Four signal channels encode meaning in both. Four surfaces provide the contexts in which all of this manifests.

01 Colour

Five construct families, three verdict states. Hue encodes type identity; saturation and weight signal runtime outcome.

02 Typography

Mono for all semantic names. Sans for prose and context. Weight and style carry constraint and modification status.

03 Form

Stable base shapes encode construct type. Modifier signals — border weight, fill, dash, outline — layer additively without displacing the base.

04 Layout

Adjacency, ordering, and depth encode relationship, sequence, and structural hierarchy across all surfaces.

Semantic Identity

Each construct carries a stable type classification that persists regardless of surface. Structure is always indigo, State always violet, Event always cyan, Data always steel, Rule always amber. Moving from code editor to diagram to data form to timeline, the same construct speaks in the same visual register — the medium adapts, the identity does not.

Authored vs Runtime Meaning

Definition captures what a construct is — its place in the governed schema, independent of any live instance. Runtime captures what is happening to it now — current lifecycle position, field values, evaluation results. These two readings are formally distinct. The same construct identity underlies both; the register changes, the identity holds.

Additive Modifiers & Runtime Overlays

Runtime state and constraint status are expressed as signals layered onto the base depiction — never as replacements. A blocked transition marks the event; it does not erase it. A violated rule flags the field row; it does not suppress it. Evaluation output — allowed, blocked, warning — is drawn on top. Overlay reveals; it does not displace.

Cross-Surface Translation

Moving from code editor to diagram to data form to timeline, a construct adopts different visual conventions appropriate to each medium. What remains invariant is its semantic identity: colour family, name form, and the modifier logic that applies to it. The system defines what stays constant so nothing needs to be relearned as surfaces change.

Allowed

The event would transition successfully with the current data and state. Runtime-only — never an authored construct property.

The event would proceed but with a condition worth noting — an unmatched guard, a near-boundary value, a missing optional field.

05 · Typography

Two faces. One rule each.

Cascadia carries all structural identity. Inter carries reading prose. No serifs. Italic marks semantic pressure, not emphasis.

Cascadia — identity & structure

Used for: wordmark, section headings, labels, chips, nav, construct names, table headers, eyebrows, inline tokens, and all code specimens. Cascadia is the face of governed structure. When text needs to read as authored or systematic, it is in Cascadia.

Inter — reading prose

Used for: body paragraphs, concept descriptions, surface notes, and any text longer than a label that needs sustained reading. Inter is engineered for screen at 15–18px and sits beside Cascadia without competing. Segoe UI Variable and Segoe UI are the system fallbacks on Windows.

Precept governs the integrity of a business entity's data across its lifecycle. The semantic visual system governs how that governance is communicated on screen.

Wordmark treatment

The identity display — Precept in Cascadia small caps — is reserved for the hero moment and deliberate brand callouts. Inline prose uses plain title case. Nav and labels are already all-caps mono and need no additional treatment. Scarcity is what makes the wordmark land.

Italic — semantic pressure only

Italic belongs to constrained states, constrained events, invariant-guarded data names, and rule-adjacent annotation in code specimens. It marks that something is under explicit rule attention. It is not used for general emphasis, decorative headings, or editorial flourish. If a reader cannot trust italic to mean pressure, the second semantic channel has been wasted.

Rule

Italic means pressure.

Not emphasis. Not decoration. Italic communicates that a construct is under rule attention. Spend it only where that meaning is true.

Verdicts · Runtime only

Emerald, coral, and yellow appear only in runtime surfaces: the data form and state diagrams. Never in syntax highlighting or static surfaces. The separation ensures these hues carry a single clear meaning — the engine evaluated your inputs and here is the result.

Rule

Every color has a fixed meaning.

Nothing is aesthetic. If a new surface uses color, it must map to an existing semantic family. Decorative use of any palette color corrupts the signal.

Rule

Color says kind. Italic says constrained.

Constraint pressure is expressed through typography, not extra hues. Constrained states, events, and guarded fields are italic within their own family color — same hue, italic only.

Rule

Verdicts are runtime only.

Emerald, coral, and yellow appear only when the engine has evaluated something. They are never used in static surfaces, syntax highlighting, or any authoring-time annotation.

Rule

Comments are outside the palette.

Dusk #9096A6 in italic keeps comments legible but structurally separate from the five semantic families. Never repurpose a family color for editorial annotation.
