Semantic Visual System

The semantic visual system defines how Precept means things on screen. It gives states, events, data, overlays, and runtime feedback a consistent visual language so the same signal reads the same way across operational surfaces like authoring, diagrams, and preview, and across explanatory surfaces like docs, README examples, and the public website. The result is faster recognition, lower cognitive load, a cleaner distinction between operational behavior and editorial explanation, and a more beautiful, more unmistakably Precept presence across every surface. People should be able to move across contexts without relearning the signal set while still feeling the product's brand confidence and craft.

Foundation rules

One system, several carriers.

A surface is the user-facing context; a carrier is the visual form that meaning takes inside it. The system keeps meaning stable across operational and editorial surfaces while allowing each carrier to adapt appropriately.

Family stability

Semantic lanes do not move.

The inherited structure, state, event, data, and rule-language families remain fixed across surfaces. Local tonal variants are allowed; family reassignment is not.

Carrier discipline

Shape outranks tint for lifecycle.

Origin, finality, and edge behavior should be expressed structurally before color is asked to do the work. This prevents diagram drift into rainbow state maps.

Constraint pressure

Typography is the second semantic channel.

Italic communicates rule pressure for constrained states, constrained events, and invariant-guarded names. The system does not create a new color family for that pressure.

Runtime separation

Verdicts remain overlays.

Emerald, rose, and amber only appear when the runtime is telling you what happened or what would happen. They are not authoring-time identities.

Primitives

These are the reusable visual parts.

Primitives are not components. They are the semantic pieces surfaces compose into their own layouts and workflows.

Structural frame

The indigo frame defines context: code block, diagram cluster, overlay shell, or diagnostics group. It tells the user where the governed information starts and stops.

Actor pairing

States and events stay visually distinct even when adjacent. The system should never make a user infer whether a label is a condition or an action from wording alone.

Data stack

Data can have tonal variation within the bright slate family, but it should still read as one factual lane. Names, types, and values may separate; the family does not.

Rule message interrupt

Gold enters the interface as an explanation the human should notice. It is an interrupt, not a running style. If too much is gold, nothing reads as rule meaning anymore.

Constraint pressure

Italic is the shared pressure signal. It should look intentional, not accidental. Surfaces may apply it differently, but they should preserve the meaning: this thing is under rule attention.

Allowed flow

Emerald flow appears when the interface is speaking about a runtime allowance or executable path. It should never be mistaken for general navigation chrome.

Overlay grammar

Status appears as a layer, not a replacement.

Overlays annotate existing semantic content. They should never erase the thing they are annotating.

Enabled overlay

Emerald can tint an edge, row accent, badge, or prospective outcome label. The state and event identities beneath it remain violet and cyan.

Blocked overlay

Rose means the path or action is structurally denied in the current runtime context. Use enough geometry or iconography that the blocked state is not color-only.

Warning overlay

Amber marks unmatched, cautionary, or attention-worthy runtime output. It should read as a diagnostic pause, not as semantic content from the DSL itself.

Minimum required rules

These rules are mandatory on every current and future surface.

These are compliance requirements, not optional style notes. A surface is incomplete until these minimum rules are true.

Preserve semantic identity

A surface may mute or intensify a family locally, but it may not swap actor families, demote rule language into generic chrome, or treat verdict colors as authoring identity.

Never ship color-only verdicts

Allowed, blocked, and warning states must use at least one non-hue carrier such as iconography, line pattern, label text, badge shape, or row position. Blocked must remain legible without hue perception.

Make current condition explicit

The current state, event availability, and runtime outcome must be named in text somewhere on the surface. Users should not need to infer the active condition from framing or tint alone.

Surface the reason when the runtime knows it

If the system knows why an action is blocked, unmatched, or exceptional, the surface must expose that cause as readable copy. Gold remains the preferred explanatory carrier when rule language is shown.

Inheritance checklist

Future surface work should start from this checklist.

This is the concise compliance split for future surface work: what must stay stable, what may adapt, and what each surface must still define locally.

Must preserve

  • Semantic family identity for structure, state, event, data, rule language, and runtime verdict overlays.
  • Minimum redundancy rules for blocked, warning, and allowed runtime states.

May adapt

  • Carrier medium: shape, weight, chip form, row styling, overlay badge, legend, or pattern.
  • Layout, density, motion, and information grouping for the local workflow.
  • Local tonal tuning within a semantic family when contrast and recognition stay intact.

Must define locally

  • Exact layout, interaction model, and density rules for that surface.
  • Focus, selection, hover, and keyboard behavior that do not collide with semantic cues.
  • Any surface-specific affordances, motion, or help text needed to make the inherited contract usable in context.
Boundary note

The system operationalizes brand meaning. It does not renegotiate it.

Use the brand spec when the question is identity, category, semantic meaning, mark logic, or typography intent. Use this document when the question is how inherited meaning shows up consistently across product surfaces and current surface applications.

Belongs in brand

Any change to inherited semantic family meaning, identity rationale, mark logic, tone of voice, or typography intent.

Belongs here

Carrier rules, overlay behavior, current surface translation guidance, and the compliance rules future surface work must preserve.

Belongs in a surface spec

Exact inspector layout, event-row density, diagram interaction states, keyboard models, motion timing, or other behavior unique to a single surface once the split is needed again.
