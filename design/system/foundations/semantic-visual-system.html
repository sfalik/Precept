Precept / Design System

Semantic Visual System

The Semantic World of Precept

Precept governs the integrity of a business entity's data across its lifecycle. The semantic visual system governs how that governance is communicated on screen: how structure, states, events, data fields, and rules each look in every surface where Precept is present.

Five authored semantic families. One separate runtime verdict register that sits on top of them without replacing them. One separate chrome/support layer that scaffolds the surface around them. A stable visual grammar that survives when the context shifts from code editor to topology diagram to data form to public documentation.

02 · Concepts

Five constructs · Two registers · Four channels

The system has four structural parts. Five semantic families form the identity substrate — the authored vocabulary that stays recognizable wherever it appears. Two depiction registers translate that identity into authored source (Definition) and rendered components (Runtime). A separate runtime verdict register reports evaluation without redefining type. A separate chrome/support layer carries canvas, panel, text, and line scaffolding so the surrounding palette can complement the semantic system without competing with it. Four signal channels encode meaning in both. Four surfaces provide the contexts in which all of this manifests.

Foundation Semantic Identity

Each construct carries a stable type classification that persists regardless of surface. Structure is always indigo, State always violet, Event always cyan, Data always steel, Rule always amber. Moving from code editor to diagram to data form to timeline, the same construct speaks in the same visual register — the medium adapts, the identity does not.

Authored vs Runtime Meaning

Definition captures what a construct is — its place in the governed schema, independent of any live instance. Runtime captures what is happening to it now — current lifecycle position, field values, evaluation results. These two readings are formally distinct. The same construct identity underlies both; the register changes, the identity holds. Verdict hues report runtime outcome in their own register rather than becoming new semantic families.

Additive Modifiers & Runtime Overlays

Runtime state and constraint status are expressed as signals layered onto the base depiction — never as replacements. A blocked transition marks the event; it does not erase it. A violated rule flags the field row; it does not suppress it. Evaluation output — allowed, blocked, warning — is drawn on top. Overlay reveals; it does not displace.

Cross-Surface Translation

Moving from code editor to diagram to data form to timeline, a construct adopts different visual conventions appropriate to each medium. What remains invariant is its semantic identity: colour family, name form, and the modifier logic that applies to it. Chrome/support tokens stay outside that identity and handle scaffolding instead, which means a surface can choose a complementary chrome palette without inventing new semantic meaning. The system defines what stays constant so nothing needs to be relearned as surfaces change.

03 · Typography

Two faces. One rule each.

Typography governs how identity, prose, and semantic pressure are read across all five constructs. Geist Mono carries structural identity. Geist carries reading prose. No serifs. Italic marks semantic pressure, not emphasis.

Primary split

Type separates authored identity from reading voice.

Typography in Precept is not a styling layer. It divides structural naming from explanatory reading, then reserves a third move for semantic pressure. The section needs to read as a system of roles, not a list of font choices.

Rule

Every color has a fixed meaning.

Nothing is aesthetic. If a new surface uses color, it must map to an existing semantic family. Local tonal variants may clarify hierarchy inside a family, but they do not create new semantic lanes. Decorative use of any palette color corrupts the signal.

Rule

Color says kind. Italic says constrained.

Constraint pressure is expressed through typography, not extra hues. Constrained states, events, and guarded fields are italic within their own family color — same hue, italic only.

Rule

Verdicts are runtime only.

Emerald, coral, and yellow appear only when the engine has evaluated something. They are never used in static surfaces, syntax highlighting, or any authoring-time annotation.

Rule

Comments are outside the palette.

Dusk #9096A6 in italic keeps comments legible but structurally separate from the five semantic families. Never repurpose a family color for editorial annotation.

Allowed

The event would transition successfully with the current data and state. Runtime-only — never an authored construct property.

Blocked

One or more rules prevent the event from proceeding. The blocking rule should be surfaced alongside the verdict, not withheld.

Warning

The event would proceed but with a condition worth noting — an unmatched guard, a near-boundary value, a missing optional field.

Note

Precept does not evaluate transitions at author time. The Submit event only executes when explicitly fired via the runtime API or MCP tool.

Tip

Use precept_inspect to preview all available transitions for a given state without modifying any data.

Caution

Field values passed to precept_fire must match declared field types. Type mismatches are caught at evaluation time, not authoring time.

Warning

Removing a state referenced by an active transition causes a compile error. All references must be updated before the precept can be re-evaluated.

Guide

Defining Precepts

A precept is a governed schema that defines what a business entity is allowed to become. It declares fields, names the states it can occupy, and specifies which events drive transitions between them. Everything the entity can do  —  and everything it cannot  —  is declared here.

States

A state is a named position in the entity’s lifecycle. Every precept has at least one state marked initial  —  this is where a new instance begins. States are declared with the state keyword and a name drawn from the State construct family.

Nothing happens to an entity unless an event fires. States are inert until an event arrives. This is what makes Precept deterministic: the current state and the fired event fully determine the outcome.

Events & Transitions

Events are the only mechanism that advances state. An event declared under a specific source state can only fire from that state. Attempting to fire it from any other state has no effect  —  Precept does not throw; it simply finds no matching transition.

What a transition block does

  • Evaluates all rule guards in declaration order  —  first failure blocks the whole event
  • Applies all set expressions to produce the next field snapshot
  • Moves the entity to the target state declared by transition to
  • Returns the outcome: allowed, blocked, or rejected
Note

Event ensures are scoped to the event that declares them. Rules declared at the precept level are evaluated on every transition, regardless of which event fired.

A minimal precept

The following declares a LoanApplication with two states, one field, and one transition. It is the smallest complete example that demonstrates the full structure.

precept LoanApplication
  field  Amount   as number

  state  Draft    initial
  state  Submitted

  on  Submit  from Draft
    rule  Amount > 0  because "Amount must be positive"
