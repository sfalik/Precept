1. Brand Identity

1.1 Positioning & Narrative LOCKED

Primary frame

Category creation — "domain integrity engine"

No competitor occupies this frame. Precept defines the category, then becomes synonymous with it. Same pattern as Temporal ("durable execution"), Docker ("containerization"), Terraform ("infrastructure as code").

Secondary frame

AI-native — "the contract AI can reason about"

Deterministic semantics + structured tool APIs make Precept uniquely suited for AI authoring and validation. This is the "why now" signal, not the primary identity.

Single-sentence positioning

"Precept is a domain integrity engine for .NET — a single declarative contract that binds state, data, and business rules into an engine where invalid states are structurally impossible."

Narrative archetype

Category Creator: "This is a new kind of thing."

Like Temporal's "durable execution" or Tailwind's "utility-first CSS." Requires category education but earns the strongest position.

README narrative arc

  1. One sentence: what it is (resolution)
  2. One sentence: why it's different (implicit conflict)
  3. One code sample (proof)
  4. Feature list or badge row (comprehensiveness signal)
The name "Precept" carries narrative weight — it means "a strict rule of action." The name IS the story.

1.2 Voice & Tone REVISED

Authoritative with warmth.

The voice states facts. It doesn't hedge. It doesn't oversell. It finds the wit in precision. When something matters, it says why — and the clarity itself can be the humor.

Models: Serilog's professionalism + MediatR's economy + Workflow Core's clarifying warmth.

  • "Precept supercharges your business logic!"
  • "Say goodbye to bugs forever!"

The three marks share one palette and one conceptual language. Each surfaces a different aspect of Precept: the state machine, the written rule, and the two ideas combined. The combined form is the primary mark. In every state-bearing mark, the canonical outer silhouette is the rounded state card; the small internal dot remains as the origin cue and centers within the state shape whenever it appears without a label. 1.4 Color System LOCKED

The Precept color system is one semantic system used across docs, UI, website, README, and product surfaces: five semantic families plus three semantic signal colors. Indigo anchors the system as the structural lane. Violet marks states, Cyan marks events, Bright Slate marks data, and Gold marks human-readable rule messages. Emerald, Amber, and Rose are reserved for shared signal meaning across surfaces. Background #0c0c0f. Surface sections may define local tonal variants for clarity, but those variants do not expand the general system.

Locked design

The semantic lanes are fixed. Surfaces may adapt shade, weight, geometry, or opacity locally, but they keep the same meanings. § 2.1 defines syntax-specific tonal variants. § 2.2 defines diagram-specific application. Neither section adds new top-level families.

Semantic color as brand

Precept uses color to say what the system means. Indigo carries the load-bearing frame; violet separates state from cyan action; bright slate keeps data factual; gold isolates human-language rule text; and Emerald, Amber, and Rose are reserved for shared signal meaning such as allowed/do/success, caution/attention, and blocked/don't/error. Meaning comes first. Aesthetics follow.

2.2 State Diagram LOCKED

Purpose

The VS Code preview panel state diagram. Renders the precept as a diagrammatic finite-state automaton using the same semantic families defined in § 1.4. Shape carries origin/final structure; hue carries category.

Visual application

The state diagram is a structural reading surface first. Base silhouette explains what kind of state is being shown. Edge form explains what kind of event relationship exists. Runtime overlays explain what the current instance can do from here. Color confirms those readings, but the diagram should remain intelligible even before hue is read.

Single canonical silhouette: States are not tinted by role (initial/intermediate/terminal). The rounded state card is the base silhouette everywhere in the diagram. Origin keeps a small internal dot that centers when it stands alone; if the state name shares the card, the dot shifts left into a single-line dot + label row with both elements vertically centered and never wrapping. Terminal keeps an outer echo border. All state nodes use locked indigo for borders and violet for state names. Brand-mark adaptations inherit that rounded-card silhouette; in the mark itself, the paired state shapes normalize to equal-size square cards so the icon stays compact and balanced.
Semantic visual grammar:

The diagram reads through these channels, in order: base silhouette (state card), structural marker (origin dot or terminal echo), edge grammar (transition, conditional, self-loop), runtime overlay (current tint, enabled path, blocked path), then color and label text. If a distinction can be carried by shape, border, or edge form, do not spend a new hue to express it.

2.3 Timeline DRAFT

Purpose

Horizontal event timeline in the VS Code preview panel. It reads left-to-right as a simple progression: origin, committed history, current state, then next possible actions. The timeline is the clearest answer to two questions: how did this instance get here, and what can happen next?

Visual application

Concept 28 locks the timeline around one quiet card family and a fixed reading order. Position explains time first. Card role explains whether the viewer is looking at state, event history, or a next action. Runtime overlays explain whether something is current, enabled, blocked, or terminal. Color supports those readings, but it does not replace them.

Authoritative rule:

The timeline is not an audit table. Show only the minimum needed to read sequence and act: one origin card, compact past cards, one present anchor, and future action lanes that expose the next real choices. If a detail does not explain history, current position, or next action, it belongs in the data panel.

Semantic visual grammar:

The timeline reads through stacked channels, in order: placement (past, present, future), base silhouette (state card or event card), structural markers (origin dot, terminal echo), adjacency (event paired with target state), runtime overlay (present tint, blocked wash, enabled accents), then color and copy. If a meaning can be carried by an earlier channel, do not invent a later one.

2.4 Data Panel DRAFT

Purpose

The field inspector surface in the VS Code preview panel. Displays all precept fields in a responsive grid, supports direct editing of fields declared editable (via in State edit Field), and shows constraint status (invariants and state assertions) in a bottom strip. The data panel is where abstract data declarations become concrete runtime values.

Visual application

Concept 28 locks the data panel around one stable field-unit anatomy. The unit stays constant whether a field is passive, editable, dirty, valid, invalid, true, false, or changed. Runtime meaning arrives through overlays, emphasis shifts, and grouped validation context. Color reinforces those states, but the panel should remain legible even before hue is read.

Authoritative rule:

The data panel is a readable field surface first, not a form-builder skin. Every unit should preserve the same anatomy: label row, control/readout, one supporting note line, and optional validation grouping beneath. Runtime states alter tint and border, but they do not break the unit grammar.

Semantic visual grammar:

The data panel reads through these channels, in order: unit anatomy (label, value, note), control mode (readout or input), local emphasis (focus ring, dirty echo, invalid border), group context (validation strip and field-target pills), then color and copy. Editing is a capability overlay on the field unit, not a different component family.

2.5 Inspector Panel LOCKED

Purpose

Runtime verdict surface. The data/field inspector in the preview webview, showing live instance state during event firing. Developers inspect field values, constraint violations, and the effect of transitions. The panel is where data becomes concrete — where abstract rules meet runtime reality.

Primary visual concerns

  • Field visibility and hierarchy — which fields are set, which are null/default, which are guarded by invariants
  • Constraint feedback — if a field violates an invariant, the violation must be obvious without requiring text parsing
  • Event argument preview — when firing an event, the panel shows what arguments are available and their values
  • State transition clarity — before and after state, with visual indication of which fields changed

Color application

Inspector uses the data and verdict color systems:

  • Field names: Slate #B0BEC5, normal or italic if guarded by invariant. Same as syntax highlighting so the reader recognizes the field
  • Field types: Slate #9AA8B5. Type info is secondary to the name and value
  • Field values: Slate #84929F for normal values, or verdict colors if a constraint is violated
  • Current state label: Violet #A898F5 — matches syntax highlighting and diagram state naming
  • Constraint violation indicators: Blocked #FB7185 for rejected events or invariant violations. Verdict colors signal runtime constraint feedback
  • Enabled indicators: Enabled #34D399 for valid state, successful transitions, or non-violated fields
  • Read-only field indicators: Dusk indigo #9096A6 or a neutral shade for fields not declared editable in the current state via in State edit Field

Typography application

Cascadia Cove monospace, 12–13px for field names and values:

  • Field name: Normal weight, 12px, left-aligned. Field names that are invariant-guarded can be italic
  • Field value: 12px, monospace to match code appearance. Wider line-height (1.6–1.8) for readability in a tabular layout
  • Constraint message: 11px, gold #FBBF24 text below the violated field. Message text is the human explanation, so it earns color emphasis and smaller size to distinguish from data values

Accessibility notes

  • Color + icon redundancy: Constraint violations use both color (Blocked rose) and an icon (✗ or warning symbol). A color-blind developer still sees the symbol
  • Table structure: If inspector data is rendered as a table (field name | type | value), use proper <th> headers and <tr>/<td> semantics so screen readers announce the structure
  • Formal accessibility audit (color-blind simulation, screen reader testing) is a backlog item

AI-first note

The inspector panel is JSON-serializable state data. An AI agent reading the inspector output needs to understand: current state, current field values, which constraints are violated, and what the next transition would do. Structured layout with consistent field ordering and color-coded constraint feedback means the agent can parse violations without string matching on error text. The inspector is both a human debugging tool and an API contract for AI tools.

2.6 Docs LOCKED

Purpose

Internal documentation artifacts (design docs, tutorials, reference material). These are team-facing documents, not a public-facing documentation site. Where developers learn Precept concepts, read language design docs, explore sample files, and navigate API reference.

Primary visual concerns

  • Hierarchy and scanability — tutorials, reference, samples, and theory must be visually stratified so a developer finds what they need instantly
  • Code sample rendering — .precept files embedded in docs must use the same syntax highlighting colors as the editor, so readers feel continuity
  • Conceptual callouts — "this is a precept" (small caps), "this is a pattern" (styled differently from narrative prose)

Color application

Docs maintain the dark-mode semantic palette but may introduce additional neutral/gray shades for navigation, backgrounds, and UI chrome:

  • Code blocks embedded in prose: Use the full syntax highlighting palette — indigo for structure, violet for states, cyan for events, slate for data, gold for messages. Readers see the same colors as in their editor
  • Brand color accents: Navigation, headings, and link highlights use deep indigo #6366F1. This keeps the brand color family visible throughout the docs without overwhelming
  • Callout boxes: Locked brand decisions, conceptual definitions, and "Precept principle" statements may use subtle indigo background (current practice in brand-spec.html callout style)
  • Verdict colors in diagrams and reference tables: If diagrams are embedded in docs, use the same color system as the preview webview (§2.2)
  • Neutral grays for structure: Navigation backgrounds, borders, table headers, and dividers use the muted palette (#27272a, #18181b, #0c0c0f) to avoid competition with semantic colors

Typography application

Cascadia Cove for code, primary brand font for headings and UI:

  • Page headings (H1): Cascadia Cove 700, small caps, 0.1em letter-spacing — same wordmark treatment as README and brand-spec
  • Section headings (H2, H3): Cascadia Cove 600–700, normal case. Hierarchy through size and weight, not typeface
  • Prose: System font stack or Cascadia Cove 400, 14–16px line-height 1.6–1.8. Readable for long-form content
  • Code samples: Cascadia Cove 400, 12–13px with 1.8 line-height. Syntax highlighting colors applied

Accessibility notes

  • Heading hierarchy: H1 → H2 → H3, no skips. Screen readers rely on hierarchy to navigate long docs
  • Code sample alt text or captions: Embedded .precept samples should have a descriptive caption or nearby prose explaining the concept they demonstrate
  • Light theme: Not planned. Docs are dark-mode-only, matching the editor experience. Light theme support is a backlog item.

AI-first note

The docs are where AI agents gather context before writing Precept. Claude agents reading the tutorial section learn syntax and patterns. Reference docs with clear structure (headings, code blocks in <pre>, definition lists) are easier for agents to parse. Embedded .precept samples double as training data — the agent can read a sample and a caption and learn a pattern. Docs design matters as much for AI comprehension as human comprehension.

2.7 CLI / Terminal ASPIRATIONAL

⚠️ CLI surface is planned; not yet implemented. A standalone precept CLI tool does not currently exist. PRECEPT diagnostic codes (PRECEPT001–PRECEPT054) are surfaced in VS Code's Problems panel via the language server, not in terminal output. dotnet build uses standard MSBuild formatting. This section defines a design contract for a future CLI tool.

Purpose

Command-line output, build logs, diagnostic messages, and error reporting for a future precept CLI tool. When implemented, developers would see this surface when running diagnostics, validation, or compilation from the terminal. This is where the brand system meets terminal constraints.

Primary visual concerns

  • Error signal clarity — a developer must instantly see "this is an error" without reading prose
  • Terminal color support — many terminals support 256-color or true color; output must be legible in both dark and light terminal themes
  • Structured output — diagnostic messages, file paths, line numbers, and error text must separate visually even in monospace
  • No ambiguity with success/failure — the verdict colors (emerald/rose/amber) must be reserved for actual success/failure signals, not decorative accents

Color application

CLI output uses verdict colors strategically and sparingly:

  • Success messages: Enabled #34D399 — "Build succeeded" or "✓ All checks passed"
  • Error messages: Blocked #FB7185 — "Error: Invalid syntax at line 12" or "✗ Constraint violated"
  • Warnings: Warning #FCD34D — "Warning: Event 'Activate' has no guard; all paths succeed"
  • Neutral output (file paths, context info): Default terminal color (no color applied). The terminal's default foreground color is most readable across theme variations
  • Emphasis (bold text): File names, identifiers, and code elements can use bold or backtick quoting to stand out. Avoid color for emphasis on the CLI
  • No structural indigo on CLI. Deep indigo #6366F1 does not render well on light terminal themes and may be illegible. Reserved for UI only

Typography application

Monospace rendering, standard terminal fonts:

  • Error/warning headers: Bold + verdict color, e.g., ✗ ERROR: (bold rose) followed by the message
  • File path and line number: Normal weight, default color, e.g., file.precept:12:5: (no color formatting)
  • Diagnostic message: Normal weight, default color. If a message quotes code, use backticks: `FieldName` or double-quotes: "message text"
  • Suggestion or hint: Dim/muted color if the terminal supports it; otherwise, normal color with a → prefix

Accessibility notes

  • Color + symbol redundancy: Errors use both color (rose) and a symbol (✗). Warnings use both color (amber) and a symbol (⚠ or ⚡). A developer with red/green color blindness still sees the symbol
  • Light terminal compatibility: Verdict colors (#34D399, #FB7185, #FCD34D) are calibrated for dark backgrounds. For light terminals, either detect theme and swap to darker shades, or remove color and rely on symbols only
  • CLI color audit: Current CLI tool color application needs an audit pass to ensure consistency with these principles. This is a backlog item.

AI-first note

CLI output is often captured by AI tools and parsed as structured data. If a future CLI tool provides JSON output, that would be machine-readable; human-readable error messages should be parseable too. Consistent error formatting (error symbol + colon + error code + message) means an AI agent can regex-parse failures. Color codes in terminal output can be captured or stripped by tools, so color should never be the only signal — symbol and text structure matter more.

2.8 Canonical Hero Snippet TEMPORARY

Status note: This section captures the current README "Quick Example" as the reusable hero artifact for README, VS Code Marketplace, NuGet, Claude Marketplace, and AI-facing contexts. The current temporary live hero is the top-rated Subscription Billing candidate from the visual language exploration. It remains TEMPORARY until the final hero decision confirms whether this domain stays or is replaced.

What this section locks

This section locks two things: the exact text of the current hero sample and the rules for reusing it without drift. The live README remains the public surface; this is the brand-facing copy-and-guidance reference that keeps other surfaces in step with it.

Canonical text to reuse

The Contract (DSL) — 21 lines. Copy verbatim when a surface uses the DSL block. Preserve the blank lines and the indented transition steps; they are part of the pattern legibility.

precept Subscription

field PlanName as string nullable
field MonthlyPrice as number default 0

invariant MonthlyPrice >= 0 because "Monthly price cannot be negative"

state Trial initial, Active, Cancelled

event Activate with Plan as string, Price as number
on Activate assert Price > 0 because "Plan price must be positive"

event Cancel

from Trial on Activate when PlanName == null
  -> set PlanName = Activate.Plan
  -> set MonthlyPrice = Activate.Price
  -> transition Active
from Active on Activate -> set MonthlyPrice = Activate.Price -> no transition
from Active on Cancel -> transition Cancelled
from Cancelled on Activate -> reject "Cancelled subscriptions cannot be reactivated"

The Execution (C#) — pair with the DSL block on surfaces that have room for both. Copy verbatim, including the closing comment line.

var def = PreceptParser.Parse(dslText);
var eng = PreceptCompiler.Compile(def);
var inst = eng.CreateInstance(state, data);
var result = eng.Fire(inst, "Activate", args);
// result.IsSuccess, result.UpdatedInstance
