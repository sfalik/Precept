# Temporal Type Strategy — NodaTime-Aligned, Philosophy-Driven**Research date:** 2026-04-13**Author:** Frank (Lead/Architect, Language Designer)**Status:** Synthesis document — consolidates five rounds of design discussion into a unified forward-looking strategy.---## 1. Governing PrincipleNodaTime exists because `System.DateTime` lets you be implicit about what your data means. Precept exists because scattered service-layer code lets you be implicit about what your rules mean. Both libraries are responses to the same failure mode: **implicit behavior creates bugs; explicit behavior creates predictability.**NodaTime forces you to decide: is this a calendar date, a point on the timeline, a time of day, or a duration? You cannot leave it ambiguous. The type system encodes your answer and the compiler enforces it.Precept forces you to declare: what are the fields, what are the rules, what are the allowed transitions? You cannot leave them scattered. The contract encodes your answer and the runtime enforces it.This is not a coincidental overlap. It is the same design philosophy applied at different layers:| Precept applies it to... | NodaTime applies it to... | The shared principle ||---|---|---|| `date` over `string` | `LocalDate` over `DateTime` | Be explicit that this is a calendar date || `decimal` over `number` | `Duration` over `TimeSpan` | Be explicit about precision and semantics || `choice` over `string` | `DateTimeZone` over `string` | Be explicit about the allowed values || `integer` over `number` | `Period` over `TimeSpan` | Be explicit about what units mean || `instant` over `number` | `Instant` over `DateTime` | Be explicit that this is a point on the timeline || `timezone` over `string` | `DateTimeZone` over `string` | Be explicit that this is a timezone identifier |Precept's design principles ground this directly:- **Principle #1 (Deterministic, inspectable model):** "Fire/inspect always produces the same result for the same inputs." NodaTime's type separation makes it structurally clear which operations are deterministic (all local types, instant comparison) and which require external context (timezone conversion). The types themselves encode the determinism boundary.- **Principle #2 (English-ish but not English):** The DSL names — `date`, `time`, `instant`, `duration`, `timezone` — are English words that communicate exactly what the data is. A domain author reads `field FiledAt as instant` and understands this is a point on the timeline, not a date on a calendar.- **Principle #8 (Sound, compile-time-first static analysis):** NodaTime's type separation enables Precept's compiler to catch temporal misuse statically. `date + instant` is a type error. `instant.year` is a compile error. The types carry enough information for the compiler to reason about correctness without runtime evaluation.- **Principle #12 (AI is a first-class consumer):** Named temporal types with precise semantics give AI consumers a vocabulary to reason about entity data and generate correct precepts. An AI that sees `field DueDate as date` knows the field supports calendar arithmetic and date comparison — it does not need to infer this from naming conventions on a `string` or `number` field.The governing question for every type decision in this strategy is: **"If a domain author has this kind of data, does giving it a named type help them be explicit about what it means?"**---## 2. The Temporal Type Model### Assessment FrameworkFor each NodaTime type, the question is not "is this safe enough to expose?" — that framing leads to defensive exclusion. The question is: **"Does this type eliminate an ambiguity that would otherwise exist?"**A type earns its place in the DSL when:1. Without it, authors encode the data as a weaker type (`string`, `number`, `integer`) and lose semantic clarity.2. The type name communicates intent — a reader knows what the field holds without reading comments.3. The compiler can enforce contracts that the weaker type cannot.4. The NodaTime backing type provides correct, tested behavior for the operations the domain needs.### Type Decision Table| NodaTime Type | Precept DSL Name | What it makes explicit | Backing type | Phase | Decision rationale ||---|---|---|---|---|---|| `LocalDate` | `date` | This is a calendar date — not a timestamp, not a string that looks like a date. The Determinism Model — Reframed### Determinism is about explicit inputs, not absent dependenciesThe previous analysis framed determinism as: "remove all external dependencies from the evaluation." This led to excluding types and operations defensively — if it touches the TZ database, it's non-deterministic; exclude it.The correct framing is: **determinism means the same explicit inputs always produce the same output.** The question is not "does this operation have dependencies?" — everything has dependencies (.NET runtime, NodaTime version, CPU architecture). The question is: **"Are all inputs to this operation visible?"**Precept's determinism guarantee (Principle #1) actually states: *"Fire/inspect always produces the same result for the same inputs."* The inputs are:1. The `.precept` definition2. The current entity data (field values)3. The operation (event name + arguments, or field update)4. The runtime environment (.NET version, NodaTime version, **TZ database version**)Items 1–3 are explicitly supplied by the user or hosting layer. Item 4 is the runtime environment — the same category as the .NET version or NodaTime package version. Precept has always been deterministic *relative to its runtime environment*. This is what determinism means for every software system: same code + same data + same environment = same result.### Where temporal types sit in the determinism model| Type | Operations | External dependency | Dependency visible? | Determinism ||---|---|---|---|---|| `date` | Arithmetic, comparison, accessors | ISO calendar (fixed, never changes) | Invisible but immutable | **Full** — the calendar is a constant || `time` | Arithmetic, comparison, accessors | None | N/A | **Full** || `instant` | Comparison, duration arithmetic | None | N/A | **Full** — nanosecond comparison is pure math || `duration` | Comparison, arithmetic, accessors | None | N/A | **Full** — unit conversion is pure math || `timezone` | Equality | IANA TZ database (for validation) | Visible in the type/literal | **Full** — validation is deterministic per DB version || `datetime` | Arithmetic, comparison, accessors | ISO calendar (fixed) | Invisible but immutable | **Full** || Conversion functions | `toLocalDate`, `toInstant`, etc. | IANA TZ database (for DST rules) | **Visible in the expression** — the `timezone` parameter is an explicit input | **Full relative to runtime environment** |The key insight: **the TZ database is an explicit input to timezone conversion, not a hidden dependency.** When the author writes:```precepttoLocalDate(IncidentTimestamp, IncidentTimezone)```The timezone is right there in the expression — an explicit parameter, typed as `timezone`, visible to the reader, to the compiler, and to `precept_inspect`. Compare this to `System.DateTime.Now`, where the system timezone is invisible. NodaTime's entire design philosophy is to make these dependencies explicit. Precept's conversion functions inherit that explicitness.### The TZ database as a runtime dependencyThe IANA TZ database is bundled with NodaTime via `TzdbDateTimeZoneSource`. It is:- **Versioned:** Each release has an identifier (e.g., `2026a`).- **Stable for active jurisdictions:** The US last changed DST rules in 2007. European timezone rules have been stable for decades. Changes affect a small number of jurisdictions per update (~1–3 per release).- **Already in the dependency tree:** The moment Precept adopts NodaTime, the TZ database is present — whether Precept exposes timezone operations or not.- **Operationally standard:** Every system that handles timezone-sensitive data (banking, healthcare, aviation, logistics) already manages TZ database freshness. This is not a new burden Precept introduces.Precept could expose the TZ database version in diagnostic output (`precept_inspect` includes the TZDB version), making it as visible as the NodaTime package version. This is a transparency feature, not a safety concern.---## 5. Conversion FunctionsTimezone conversion functions are the bridge between the timeline domain (instants, durations) and the calendar domain (dates, times). They make the timezone dependency explicit — visible in the expression, typed at the call boundary, inspectable by `precept_inspect`.### Function Signatures| Function | Signature | Semantics ||---|---|---|| `toLocalDate` | `(instant, timezone) → date` | Extract the local calendar date for an instant in a timezone. "What date is it at this moment in this timezone?" || `toLocalTime` | `(instant, timezone) → time` | Extract the local time-of-day for an instant in a timezone. "What time is it at this moment in this timezone?" || `toInstant` | `(date, time, timezone) → instant` | Convert a local date + time-of-day + timezone → UTC instant. "What UTC moment corresponds to this local time in this timezone?" |### Why These Functions ExistWithout them, the multi-timezone compliance rule must be split between the precept and the hosting layer:```precept# WITHOUT conversion functions — rule is splitfield FilingDeadline as instant  # Computed by hosting layerinvariant FiledTimestamp <= FilingDeadline because "Filing deadline has passed"# The MEANING of FilingDeadline (30 days, 11:59 PM, incident timezone) is NOT in this file.```With them, the entire rule lives in one file:```precept# WITH conversion functions — rule is completeinvariant FiledTimestamp <= toInstant(    toLocalDate(IncidentTimestamp, IncidentTimezone) + days(30),    time(23:59:00),    IncidentTimezone) because "Claim must be filed by 11:59 PM local time on the 30th day after the incident"```The second form satisfies the philosophy's "one file, complete rules" guarantee. An auditor reads the precept and sees the entire business rule — 30 days, 11:59 PM, incident timezone. Nothing is hidden in the hosting layer.### DST Ambiguity ResolutionWhen `toInstant` converts a local date + time + timezone → UTC instant, DST transitions can create ambiguity:- **Gap** (clocks spring forward): The local time doesn't exist. Resolution: map to the instant *after* the gap (standard spring-forward behavior). NodaTime's `Resolvers.LenientResolver` for gaps.- **Overlap** (clocks fall back): The local time occurs twice. Resolution: map to the *later* instant (the offset after the transition). This is the safer choice for deadline calculations — it gives the later, more generous deadline.The resolution strategy is **deterministic and documented** — same inputs always produce the same output. No author choice, no hidden parameter.### ComposabilityConversion functions produce and consume existing types:- `toLocalDate` returns a `date` — all date operations work on the result.- `toLocalTime` returns a `time` — all time operations work.- `toInstant` returns an `instant` — all instant comparison and duration arithmetic work.The functions are bridges, not new types. They connect the timeline domain to the calendar domain through explicit timezone parameters. This is exactly how NodaTime itself structures conversions.---## 6. No bare integer/number arithmetic. `date + days(n)` is the only day-offset form. `days(n)` returns `period`, not `duration`. This is strictly stronger than the v1 strategy's `date + integer` rule.3. **Date subtraction:** `date - date → number` → `date - date → period`. Returns a `period` preserving structural calendar components, not a bare number or duration. Matches `Period.Between(d1, d2)`.4. **Month/year arithmetic (ships in Phase 1 with `period`):** `date + months(n) → date`, `date + years(n) → date`, `date + weeks(n) → date`. All constructors return `period`. `period` is a full surface type in Phase 1, not deferred to Phase 2.5. **Parsing:** `date(2026-01-15)` → `LocalDatePattern.Iso.Parse(value)`. Stricter ISO 8601 validation and better error messages than `DateOnly.ParseExact`.6. **Serialization:** MCP tools emit `"2026-01-15"` (ISO 8601 string). `NodaTime.Serialization.SystemTextJson` handles JSON serialization.7. **NuGet dependencies:** Add `NodaTime` and `NodaTime.Serialization.SystemTextJson` to `src/Precept/`.### What doesn't change in #26- DSL surface: `field DueDate as date` — unchanged.- Typed literal syntax: `date(2026-01-15)` — unquoted ISO content inside typed parens (Decision #18).- Comparison operators — unchanged.- Accessors: `.year`, `.month`, `.day` — unchanged (new: `.dayOfWeek`).- Constraints: `nullable`, `default`, etc. — unchanged.### New proposals needed| Proposal | Scope | Phase ||---|---|---|| `instant` type | Comparison, duration arithmetic, ISO 8601 UTC literals, no component accessors | Phase 2 || `duration` type | Field type + expression-result type, constructor functions, comparison, accessors | Phase 2 || `time` type | Time-of-day, comparison, arithmetic with `hours(n)`/`minutes(n)`, accessors | Phase 2 || `timezone` type | IANA identifier validation (compile-time + runtime), equality comparison | Phase 2 || Timezone conversion functions | `toLocalDate`, `toLocalTime`, `toInstant` — bridges between timeline and calendar domains | Phase 2 || `datetime` type | Combined date+time, comparison, decomposition accessors | Phase 3 |---## 8. This temporal type strategy is the natural consequence of applying that shared philosophy to temporal concepts.Each type in the strategy earns its place by eliminating a specific ambiguity:- `date` eliminates "is this a date or a string that looks like one?"- `time` eliminates "is this a time of day or minutes-since-midnight encoded as an integer?"- `instant` eliminates "is this a moment in time or an epoch-seconds count stored as a number?"- `duration` eliminates "is this elapsed time or an hour-count stored as an integer?"- `timezone` eliminates "is this a valid IANA timezone identifier or an arbitrary string?"- `datetime` eliminates "is this a local date-and-time or two decoupled fields pretending to be coupled?"The conversion functions make the final piece explicit: when you cross the boundary between the timeline domain and the calendar domain, the timezone is right there in the expression — not hidden in a type, not implicit in a system setting, not delegated to the hosting layer.This is what both NodaTime and Precept exist to deliver: **explicit, predictable behavior — because that is how you build systems people can trust.**
