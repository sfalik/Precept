# Dot-Access vs Function-Call Precedent Survey**Question:** Is parameterized dot-access (`value.operation(arg)`) exclusively a developer-language pattern, or do non-developer-facing DSLs use it too?**Context:** Precept currently has 18 functions (all `function(value)` form) and zero-arg dot accessors (`.count`, `.length`). The temporal type system proposal needs `inZone` — should it be `inZone(instant, tz)` or `instant.inZone(tz)`?## Summary Table| System | Target Audience | Syntax Pattern | Has Parameterized Dot-Access? ||--------|----------------|----------------|-------------------------------|| **Terraform HCL** | Infrastructure engineers | `function(args)` only | **No** || **Salesforce Formulas** | Business admins | `FUNCTION(args)` only | **No** || **Power Automate** | Citizen developers | `function(args)` only | **No** || **OPA / Rego** | Policy authors | `function(args)` for operations; dot for data navigation | **No** (dot is navigation-only) || **Google Sheets** | End users | `FUNCTION(args)` only | **No** || **Cedar (AWS)** | Policy authors | Both constructor-style AND method-style | **Yes** — extensively || **Elixir** | Developers | Pipe operator `value \|> function(args)` | **No** (functional; no dot methods) || **Kotlin** | Developers | Extension functions: `value.function(args)` | **Yes** — first-class pattern || **NodaTime (.NET)** | Developers | Instance methods: `instant.InZone(zone)` | **Yes** — native OOP pattern |## Detailed Findings### Category 1: Non-Programmer / Low-Code DSLs#### Terraform HCL- **Audience:** Infrastructure engineers (technical but not application developers)- **Pattern:** Purely `function(args)` — 80+ built-in functions, zero dot-method syntax- **Examples:** `timeadd(timestamp, duration)`, `max(5, 12, 9)`, `replace(string, search, replace)`, `upper(string)`, `concat(list1, list2)`- **Temporal operations:** `timeadd("2017-11-22T00:00:00Z", "10m")` — function form- **Verdict:** Function-call only. No dot-access of any kind in expressions.#### Salesforce Formula Fields- **Audience:** Business admins, Salesforce admins (explicitly non-developer)- **Pattern:** Purely `FUNCTION(args)` — 100+ functions, zero dot-method syntax- **Examples:** `ADDMONTHS(date, n)`, `DAY(date)`, `CONTAINS(text, searchText)`, `IF(cond, trueVal, falseVal)`, `YEAR(date)`, `DATEVALUE(text)`, `ROUND(number, digits)`- **Temporal operations:** `ADDMONTHS(StartDate, 6)`, `YEAR(CreatedDate)`, `DAY(CloseDate)` — all function form- **Verdict:** Function-call only. Dot notation used only for field references (`Account.Name`), never for operations.#### Power Automate / Azure Logic Apps Expression Language- **Audience:** Citizen developers, business automation users- **Pattern:** Purely `function(args)` — 100+ functions, zero dot-method syntax- **Examples:** `addDays(timestamp, 10)`, `convertTimeZone(timestamp, source, dest)`, `contains(collection, value)`, `concat(str1, str2)`, `toLower(text)`- **Temporal/timezone operations:** `convertFromUtc(timestamp, 'Pacific Standard Time')`, `convertTimeZone(timestamp, sourceTimeZone, destinationTimeZone)` — **this is the exact `inZone` analog and it uses function-call form**- **Verdict:** Function-call only. The closest analog to Precept's `inZone` uses `convertTimeZone(ts, src, dest)` form.#### Google Sheets- **Audience:** End users, knowledge workers- **Pattern:** Purely `FUNCTION(args)` — `SUM(A1:A10)`, `IF(condition, true_val, false_val)`, `VLOOKUP(key, range, col, match)`- **Verdict:** Function-call only. No dot-access.### Category 2: Policy / Rule DSLs#### OPA / Rego- **Audience:** Policy authors (DevOps/security engineers)- **Pattern:** `function(args)` for all built-in operations; dot only for data/document navigation- **Examples:** `count(x)`, `split(s, ".")`, `startswith(k, "f")`, `time.now_ns()`- **Dot usage:** `input.request.kind`, `site.name` — navigation only, never parameterized- **Verdict:** Function-call for operations. Dot reserved for data navigation. No parameterized dot methods.#### Cedar (AWS) — CRITICAL FINDING- **Audience:** Policy authors (security/IAM engineers — technical but not application developers)- **Pattern:** Uses **both** constructor-style and method-style syntax, explicitly documented as two call styles- **Constructor-style (function form):**  - `ip("127.0.0.1")`, `datetime("2024-10-15")`, `decimal("1.23")`, `duration("2h30m")`  - Used for type construction / parsing- **Method-style (parameterized dot-access):**  - **Decimal comparisons:** `decimal("1.23").lessThan(decimal("1.24"))`, `decimal("1.23").greaterThanOrEqual(decimal("1.24"))`  - **Set operations:** `[1,2,3].contains(1)`, `a.containsAny([b, c, d])`, `a.containsAll([b, c])`, `[].isEmpty()`  - **IP operations:** `ip("127.0.0.1").isIpv4()`, `ip("192.168.0.1").isInRange(ip("192.168.0.1/24"))`  - **Datetime operations:** `datetime("2024-10-15").offset(duration("1h"))`, `datetime(...).durationSince(datetime(...))`, `datetime(...).toDate()`, `datetime(...).toTime()`  - **Duration conversions:** `duration("1d").toMilliseconds()`, `duration("1d").toHours()`, `duration("1d").toDays()`  - **Tag operations:** `entity.hasTag(expr)`, `entity.getTag(expr)`- **Verdict:** Cedar is a non-developer-facing policy DSL that uses parameterized dot-access **extensively** — for decimal comparisons, set membership tests, IP range checks, datetime arithmetic, and duration conversions. This directly contradicts the hypothesis that "dot-with-args is exclusively a developer-language pattern."### Category 3: Developer Languages (for reference)#### Kotlin — Extension Functions- **Pattern:** Extension functions appear as dot-access methods: `"hello".truncate(15)`, `list.endpoints()`, `5.isInRange(1, 10)`- **Insight:** Kotlin explicitly made dot-access a **retrofit pattern** — you can add `value.method(args)` syntax to any type without modifying the type. This is philosophically similar to what Precept would do with type-owned operations.#### Elixir — Pipe Operator- **Pattern:** Functional language uses pipe `|>` as the "left-to-right" alternative: `value |> String.trim() |> String.upcase()`- **No dot methods.** Dot is for module access (`String.trim`), not method dispatch.- **Insight:** Elixir proves that "value-first" reading order doesn't require dot-access — pipe achieves the same left-to-right flow without OOP method syntax.#### NodaTime (.NET)- **Pattern:** Standard OOP instance methods: `instant.InZone(zone)`, `instant.Plus(duration)`, `instant.WithOffset(offset)`, `instant.InUtc()`- **`InZone` specifically:** `public readonly ZonedDateTime InZone(DateTimeZone zone)` — parameterized dot-access on Instant- **Insight:** NodaTime is the backing library for Precept's temporal types. Its native pattern is `instant.InZone(zone)`.## Analysis### The Cedar PrecedentCedar is the most significant finding. It is:1. **A policy DSL** — not a general-purpose programming language2. **Targeting non-developer policy authors** (IAM engineers, security teams)3. **Backed by formal verification** — it cares deeply about semantic clarity4. **Explicitly dual-syntax** — it formally documents both "constructor-style" and "method-style" function calls in its language referenceCedar's design choice is deliberate: **constructors use function form, operations on values use method form.** This maps cleanly to a potential Precept pattern:- `inZone(instant, tz)` would be constructor/function style- `instant.inZone(tz)` would be method/operation styleCedar's datetime operations (`datetime.offset(duration)`, `datetime.durationSince(datetime)`) are structurally identical to what `instant.inZone(timezone)` would look like in Precept.### The Non-Developer DSL SplitThe evidence reveals a clear split among non-developer-facing systems:**Function-only:** Terraform HCL, Salesforce, Power Automate, Google Sheets, OPA/Rego- These systems target **the broadest non-developer audiences** (business admins, citizen developers, end users)- They uniformly avoid dot-access for operations**Dot-access present:** Cedar- Targets a **more technical non-developer audience** (policy/security engineers)- Uses dot-access specifically for **type-owned operations** (operations that semantically belong to a value)### The Distinguishing FactorThe pattern that emerges is not developer vs. non-developer, but **audience technical comfort level**:| Comfort Level | Systems | Pattern ||---------------|---------|---------|| Spreadsheet users / business admins | Sheets, Salesforce, Power Automate | Function-only || Technical non-developers (infra/policy) | Terraform, OPA/Rego | Function-only (but closer to the boundary) || Technical non-developers (formal policy) | Cedar | Both — with clear semantic separation || Developers | Kotlin, NodaTime, C# | Dot-access native |Cedar sits at the boundary, and its answer is: **use both, with semantic meaning.** Constructors parse/create values; methods operate on existing values.### Relevance to the Power Automate AnalogPower Automate's `convertTimeZone(timestamp, sourceTimeZone, destinationTimeZone)` is the closest non-developer analog to Precept's `inZone`. It uses function-call form. However:- Power Automate has **zero** dot-access methods for anything- It targets a **less technical** audience than Precept- It has no type system that would benefit from type-owned discovery### Implications for Precept1. **Parameterized dot-access is NOT exclusively a developer pattern.** Cedar proves that a non-developer-facing DSL can use it successfully, with clear documentation and semantic intent.2. **The precedent is conditional.** Cedar uses dot-access for **type-owned operations** where the operation semantically belongs to the value — not as a general pattern for all functions.3. **Precept's audience aligns closer to Cedar's than to Salesforce's.** Precept targets business analysts and domain experts who are comfortable with structured rules — similar to Cedar's policy authors.4. **The IntelliSense argument is strengthened.** Cedar's choice supports the idea that `.` as a discovery trigger for type-owned operations adds value for the target audience.5. **The "both" approach has formal-language precedent.** Cedar's explicit dual syntax (constructor for creation, method for operations) gives intellectual cover for Precept to do the same — keeping `function(value)` for transformations and adding `value.operation(arg)` for type-owned behaviors.6. **The broadest non-dev DSLs (Salesforce, Sheets, Power Automate) are function-only.** If Precept wants to stay maximally accessible to non-technical users, function-only remains the safest choice.
