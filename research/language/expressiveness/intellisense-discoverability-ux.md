# IntelliSense Discoverability: Dot Access vs Function Catalogs — UX Research**Author:** Elaine (UX Designer)**Date:** 2026-04-15**Requested by:** Shane**Context:** Temporal type system — should `inZone` be a function `inZone(instant, tz)` or a dot accessor `instant.inZone(tz)`?---## 1. Progressive Disclosure in Authoring Tools: "Type Then Discover" vs "Know Then Type"### The Two Patterns| | Dot Access (Type Then Discover) | Function Catalog (Know Then Type) ||---|---|---|| **Entry point** | User types a field name they already know, presses `.`, sees available operations | User starts typing a function name from memory or scans a list || **Cognitive direction** | Data-outward: "I have this field; what can I do with it?" | Operation-inward: "I need this operation; what field does it take?" || **Discovery mechanism** | Contextual menu appears after `.` — zero browsing needed | Flat or categorized function list — requires scanning or partial recall || **Progressive disclosure** | **Strong.** The dot is a natural disclosure trigger. The user's first action (naming their data) produces a scoped, relevant list of operations. Secondary operations are hidden until the context calls for them. | **Weak.** All functions are visible at once (or require the user to know a category). There is no progressive mechanism — the user sees everything or nothing. |### What UX Research Says**Progressive disclosure** (Nielsen, 2006) recommends showing users a few important options initially and deferring secondary features to a secondary screen/interaction. The dot-access pattern is a textbook implementation of this principle in a code editor:1. **Initial display:** Field names (what the user already knows).2. **Secondary display:** Operations available on that field (revealed by `.`).3. **Trigger is obvious:** The `.` character is a universally understood "open the menu" gesture in every IDE.The function-catalog pattern inverts this. It front-loads the operation names and requires the user to recall which function applies to their data. The user must hold two things in working memory simultaneously: "what is my data?" and "what functions exist?" — then match them.**Key finding:** Dot access implements two-level progressive disclosure. Function catalogs are flat — they trade discoverability for familiarity only when the user already has a mental model of the function set.---## 2. IDE/Editor IntelliSense UX Patterns### VS Code / TypeScript / C# / Python — The "Hybrid" StandardProfessional IDEs universally use **both patterns simultaneously**, with a clear split:| Pattern | When It Triggers | What It Shows ||---|---|---|| **Dot completion** | User types `variable.` | Members of the variable's type: properties, methods, fields || **Global completions** | User starts typing in an empty expression position | All in-scope functions, variables, keywords |In TypeScript, typing `myDate.` shows `getFullYear()`, `getMonth()`, `toISOString()`, etc. The user doesn't need to know these exist — they type their variable, press `.`, and discover the possibilities. This is the gold standard of "type then discover."Typing `Math.` shows `abs()`, `floor()`, `ceil()`, etc. — here, dot access works on a namespace/module, not an instance.**Critical observation:** In VS Code, global function completions (like standalone `Math.abs()`) require the user to recall that `Math` exists. Dot completion on instances (`myDate.getMonth()`) requires only that the user knows their variable name. The cognitive cost is asymmetric.### Power Fx / Power Apps — The "Excel-for-Non-Developers" StandardPower Fx is the most relevant comparison for Precept's audience (business analysts, not professional developers).**Power Fx uses function-form exclusively for operations.** There is no `Record.Filter()` syntax — it's `Filter(Record, ...)`. Power Fx reserves the dot operator for **property access only**: `ThisItem.Name`, `Slider1.Value`, `Color.Red`.However, Power Fx has a massive advantage in function discoverability that Precept does not share: **it has over 200 functions organized in a rich, categorized, searchable formula bar.** The formula bar is not just a text editor — it's an integrated reference with categorized browsers, inline documentation, and parameter hints. The sheer scale of the function library justifies this investment.**Key Power Fx IntelliSense behaviors:**- Typing in the formula bar shows a filtered dropdown of all matching functions.- Selecting a function shows parameter hints inline.- Dot access after a record/table shows only the fields of that record — never functions or methods.- The separation is absolute: **dot = data access, parentheses = function call.**### Salesforce Formula EditorSalesforce formula fields use **function-only** syntax: `IF(Amount > 1000, "Large", "Small")`, `YEAR(CreatedDate)`, `TEXT(StageName)`. No dot access for operations.Discoverability relies on:- A categorized function browser panel beside the editor.- Inline function documentation when a function is selected.- The function browser groups functions by category: Date/Time, Logical, Math, Text, etc.The audience (Salesforce admins) matches Precept's audience profile. Salesforce chose function-form because it maps to the Excel mental model their users already have.### Notion Formula EditorNotion uses `prop("Field Name")` to reference fields and functions like `dateAdd(prop("Due Date"), 30, "days")`. No dot access at all.Discoverability is extremely weak — users must consult documentation or memorize function names. Notion partially compensates with a sidebar function reference, but it's separate from the typing experience.### Airtable Formula EditorAirtable uses function-form: `DATEADD({Due Date}, 30, 'days')`, `IF({Status} = "Active", ...)`. Field references use `{curly braces}`.Like Salesforce, Airtable includes a field browser and a function browser as sidebars. No dot completion.### Summary of Patterns Across Tools| Tool | Audience | Dot for properties? | Dot for operations? | Function form? | Function catalog UI? ||---|---|---|---|---|---|| VS Code (TS/C#) | Professional devs | Yes | Yes (methods) | Yes | Global completions || Power Fx | Business/citizen devs | Yes (data only) | **No** | Yes (exclusively) | Rich categorized browser || Salesforce | Admins | No | No | Yes (exclusively) | Categorized sidebar || Notion | Knowledge workers | No | No | Yes (exclusively) | Sidebar reference || Airtable | Business users | No | No | Yes (exclusively) | Categorized sidebar || **Precept (current)** | Business analysts | Yes (`.count`, `.length`) | No | Yes (flat keyword list) | **None** |**Critical gap for Precept:** Every function-first tool that targets non-developers compensates with a rich function browser/catalog UI. Precept has none — its functions appear in a flat keyword completion list alongside keywords like `set`, `transition`, `reject`, etc. Without a categorized browser, the function-form pattern loses its primary discoverability mechanism.---## 3. Recognition Over Recall (Nielsen's Heuristic #6)### The Heuristic> "Minimize the user's memory load by making elements, actions, and options visible. The user should not have to remember information from one part of the interface to another."— Jakob Nielsen, Heuristic #6 (1994, updated 2020)Nielsen/Norman Group (Budiu, 2024) explains the cognitive mechanism:- **Recognition** provides rich context (cues) that spread activation to related information in memory, making retrieval easy. A menu of options is recognition-based.- **Recall** provides minimal cues, forcing the user to generate the answer from scratch. A blank text field is recall-based.- Recognition is **always easier** than recall because it involves more cues.### How It Applies to Dot vs Function| | Dot Access | Function Catalog ||---|---|---|| **Recognition load** | **Low.** User types a field name (already known), sees a context-scoped menu of valid operations. Every item in the menu is relevant to the current field type. | **Medium-High.** User must recall that a function exists, or scroll through a list that includes functions irrelevant to their current field type. || **Cue richness** | **High.** The field name itself is the first cue. The dot triggers a second cue (the scoped menu). The user recognizes the operation they want from a short, relevant list. | **Low.** The user starts from a blank cursor or a generic keyword list. They must generate the function name from the goal ("I want to know what timezone this is in") rather than from the data ("I have an instant; what can I do?"). || **Filtering** | **Automatic.** Only operations valid for the field's type appear. An `instant` field shows `inZone()`, not `toLower()`. | **Manual.** The user must mentally filter the full function list to find temporal functions. With 18 functions today (and growing), this is a meaningful cognitive tax. |### The Verdict on Heuristic #6Dot access is unambiguously better for Recognition Over Recall. It transforms the function discovery task from **recall** ("What function converts an instant to a zoned datetime?") to **recognition** ("Ah, `inZone` — that's the one I want" — seen in a context-scoped list after typing `FiledAt.`).---## 4. Hybrid Patterns: Dot for Properties + Functions for Parameterized Operations### Who Does This Successfully?**TypeScript/JavaScript.** The language uses dot for both property access (`obj.name`) and method calls (`obj.toString()`). There is no cognitive overhead about "when to use dot vs function" because the rule is simple: **everything goes through the dot.** The distinction is syntactic (parentheses = call) not navigational (dot vs standalone).**C#/Java.** Same pattern as TypeScript. `myList.Count` (property) and `myList.Contains(x)` (method) both go through dot. Users never wonder "should I use a member or a standalone function?" — the answer is always "type your object, press dot, find the operation."**Python.** Mixed. Some operations are methods (`"hello".upper()`, `mylist.append(x)`), others are built-in functions (`len(mylist)`, `abs(x)`, `max(a, b)`). This split is a frequent source of confusion for beginners. "Is it `mylist.length` or `len(mylist)`?" is a classic Python FAQ.**Excel/DAX.** Purely function-form. No dot access. But Excel compensates with decades of user familiarity and a massive function browser.### The Cognitive Cost of MixingThe Python example is instructive. Python's hybrid model — some operations as methods, some as standalone functions — is consistently cited as a learnability problem. Beginners must memorize which pattern each operation uses. There's no discoverability shortcut — you can't type `mylist.` and see `len` because `len` is a standalone function.**Key finding:** A hybrid where *zero-arg* accessors use dot (Precept's current `.count`, `.length`, `.year`) and *parameterized operations* use function form creates the same two-bucket confusion as Python. The user must learn: "properties are dot, operations are functions." This is learnable but adds a rule to internalize.A hybrid where **everything on a type goes through dot** (the TypeScript/C#/Java pattern) eliminates this rule entirely. Dot is the universal discovery mechanism. Functions without a receiver (standalone math: `abs()`, `min()`, `max()`) stay as standalone functions.---## 5. Discoverability in Small vs Large Function Sets### The Scale ArgumentThe function-catalog argument depends critically on catalog size:| Function Count | Catalog UX | Discovery Strategy ||---|---|---|| **5–20** (Precept today: 18) | Not a catalog — it's a list. Too small to browse meaningfully. Too small for categorization to help. | Users memorize the list, or rely on autocomplete filtering as they type. || **50–100** (Power Fx core) | Browsable with categories. Categories start to earn their cognitive cost. | Categorized sidebar + search becomes valuable. || **250+** (DAX, Excel) | Rich catalog with search, categories, documentation, examples. The catalog IS the discovery mechanism. | Users navigate the catalog like a reference book. |**Precept's 18 functions are below the threshold where a catalog provides value.** There aren't enough functions to warrant categorization, a browser sidebar, or a search mechanism. At 18, the "catalog" is just a flat list that the user must scan.Conversely, **18 functions are above the comfortable memorization threshold for non-developers.** Miller's 7±2 (1956) suggests that the typical person can hold 5–9 items in working memory. A business analyst who uses Precept occasionally cannot reliably recall 18 function names.This creates a discoverability dead zone: too many functions to memorize, too few to build a browsable catalog.**Dot access solves this dead zone.** Instead of the user needing to recall from a set of 18 unscoped functions, they type a field name and see only the 2–4 operations valid for that field's type. The cognitive load drops from 18 to ≤4 — well within the comfortable recognition range.### As the Function Set GrowsThe temporal type system adds 3 timezone conversion functions (`toLocalDate`, `toLocalTime`, `toInstant`), bringing the count to ~21. Future growth (computed field functions, additional type operations) could push toward 30–40.At 30–40 functions, the discoverability dead zone gets worse: too many to memorize, still not enough for a meaningful categorized browser. Dot access on typed fields scales naturally — the per-type operation count stays small regardless of total function count.---## 6. Real-World IntelliSense Behavior in Low-Code Tools### Power Fx Formula Bar (Documented Behavior)From Microsoft's documentation:1. **Typing a data source or table name** followed by `.` shows the fields/columns of that data source. This is dot-for-data only.2. **Typing a function name** (e.g., `Fil`) shows a filtered dropdown of matching functions: `Filter`, `Find`, etc. Selecting a function inserts it with parameter placeholders.3. **Inside function parentheses**, context-aware completions show relevant fields, variables, and operators.4. **The formula bar's "fx" button** opens a categorized function browser with descriptions and examples.The key insight: Power Fx has **two discovery paths.** Data flows through dot. Operations flow through the function catalog + formula bar browser. Both paths are fully supported with rich UI.### What the User Actually Sees (Interaction Flow)**Power Fx user writing a date filter:**1. Types `Filter(` → sees data source completions2. Types `Inventory,` → sees field completions (dot-like behavior inside function args)3. Types `DateDiff(` → sees parameter hints: `Start`, `End`, `Units`4. If they don't know `DateDiff` exists, they click the formula bar browser, navigate to "Date and Time" → find `DateDiff`**Hypothetical Precept user writing a timezone conversion (function form):**1. Types `toLocal` → sees `toLocalDate` and `toLocalTime` in keyword list2. Selects `toLocalDate(` → types `FiledAt, PolicyTimezone)` from memory or docs3. If they don't know `toLocalDate` exists, they... scroll through the flat keyword list? Search docs?**Hypothetical Precept user writing a timezone conversion (dot form):**1. Types `FiledAt.` → sees `inZone(tz)`, `toLocalDate(tz)`, `toLocalTime(tz)` — only instant-valid operations2. Selects `toLocalDate(` → types `PolicyTimezone)` with parameter hint3. Discovery was automatic — no docs needed, no catalog browsing neededThe dot-form interaction has **one fewer cognitive step** because discovery is embedded in the typing flow.---## 7. The AI-First DimensionPrecept is AI-first. Both human developers and AI agents consume the language. How does the pattern choice affect AI consumers?| | Dot Access | Function Form ||---|---|---|| **AI autocomplete assistance** | AI can scope suggestions to the field's type — high-precision completions | AI must infer type from context to offer relevant functions — more room for error || **AI reading comprehension** | `FiledAt.toLocalDate(tz)` — the data-flow direction is explicit in the syntax | `toLocalDate(FiledAt, tz)` — requires parsing argument position to understand data flow || **AI code generation** | AI generates `Field.operation()` — type drives operation choice, reducing hallucination | AI generates `operation(Field, ...)` — must recall function name independently of type |For AI consumers, dot access provides stronger type context per token, reducing the probability of suggesting invalid operations.---## UX Recommendation for Precept### The Core Finding**Dot access is unambiguously superior for Precept's specific context.** The combination of:1. **Business-analyst audience** (non-developers who cannot be expected to memorize function catalogs)2. **No function browser/catalog UI** (and no near-term plan for one)3. **Small but growing function set** (18 today, 21+ with temporal — in the discoverability dead zone)4. **Typed fields** (instant, date, time, etc. — rich types that naturally scope operation sets)5. **AI-first design** (dot provides better type context per token)...means that dot access serves both recognition over recall AND progressive disclosure better than function form for Precept.### The Recommended Pattern**Adopt the C#/TypeScript "everything through the dot" pattern for type-scoped operations:**| Category | Pattern | Example | Rationale ||---|---|---|---|| **Zero-arg properties** (already in Precept) | `Field.property` | `DueDate.year`, `MySet.count` | No change — this already exists and works || **Type-scoped parameterized operations** (NEW) | `Field.operation(args)` | `FiledAt.toLocalDate(tz)`, `FiledAt.inZone(tz)` | Discovered via dot, scoped to type || **Standalone math/utility functions** (already in Precept) | `function(args)` | `abs(x)`, `min(a, b)`, `round(x, 2)` | No receiver type — these operate on values, not typed fields. Keep as standalone. |**The rule is simple and teachable:** *"Type a field name and press dot to see what you can do with it. Use standalone functions for math and utilities."*### Why Not Pure Function Form (the "Excel Pattern")?The Excel/Power Fx pattern works **only when paired with a rich function browser UI.** Salesforce, Power Apps, Airtable — every tool that uses function-only form invests heavily in a categorized, searchable function reference panel. Precept does not have this and should not build it to serve 21 functions. Dot access provides equivalent discoverability at zero UI cost.### Is There Cognitive Overhead from Mixing Dot + Functions?Yes, but it's minimal, because the rule decomposes along a natural boundary the user already understands:- **"My data, my field"** → dot. "What can I do with DueDate? Type `DueDate.` and see."- **"Math on numbers"** → function. "How do I round? Type `round(` and fill in args."This maps to the already-familiar boundary between "my spreadsheet cells" (data) and "formulas I apply to cells" (operations). Business analysts already think this way.### Concrete `inZone` Recommendation`FiledAt.inZone(PolicyTimezone)` — dot form.The user types `FiledAt.`, sees `inZone(tz)` alongside `toLocalDate(tz)` and `toLocalTime(tz)`, and selects the operation they need. They never need to recall that `inZone` exists as a standalone function name. They never need to search for it.---## References- Nielsen, J. (1994). "Heuristic evaluation."
